# liquid-mercury · notes

## log
- 2026-05-14: v1 — dark-mode liquid-mercury simulation. Chat request: "create a satisfying liquid mercury simulation with dark mode aesthetics." **Visual technique**: 2D metaball trick. The mercury surface is *not* drawn as solid disks — instead each particle paints a soft radial gradient (white-center → grey-mid → transparent-edge, radius 3× the particle's physics radius) onto an offscreen canvas. The canvas then applies `filter: blur(7px) contrast(24) brightness(1.05) saturate(0.6)` via CSS, which crushes the overlapping gradients into a hard chrome-edged surface that flows and merges as the particles move. When two drops approach each other their gradients overlap → the filter snaps them into one larger blob; pull them apart and they smoothly pinch off. Total cost is one canvas pass + one composited CSS filter — runs 60 fps on phones. **Render stack** (z-ordered): (1) `#shadow` — soft dark radial blob under each particle (`mix-blend-mode: multiply`, opacity 0.85) so the drops sit on the floor with weight; (2) `#meta` — the metaball gradients with the chrome filter; (3) `#sheen` — small bright specular dots at the upper-left of each drop (`mix-blend-mode: screen`, opacity 0.85) which give the mercury its characteristic shine; (4) `#scene` — full-screen cool-to-warm vertical gradient (`mix-blend-mode: overlay`, opacity 0.45) that tints the mercury's reflection from blue-sky-top to warm-floor-bottom; (5) vignette + faint scanline overlay via `.stage::before`. **Physics**: 96 default particles (slider 20–160), each with position + velocity + radius (10–16 px). Gravity at 600 px/s² scaled by a 0–1.8 slider. Damping `vx,vy *= 0.985` per frame for the heavy-fluid feel. Walls bounce with 0.55 loss and floor friction `vx *= 0.85`. Soft pairwise collisions: when two particles are closer than `0.84 · (rA + rB)` they're pushed apart by half the overlap each AND get a relative-velocity exchange clamped to `0.5×` so mercury stays cohesive (overlap left slightly so metaballs visibly merge — that's the look). O(n²) collision is fine at n≤160 and ~16 ms per frame. **Input**: pointer-down → strong attractor (1200 px/s² toward the cursor + pointer-velocity carried into the drops at 8% — flick the cursor and the mercury shoots that way). Without holding, a Magnet toggle in the dock turns on a softer attractor at 380 px/s² over a 240 px radius. Pointer velocity tracked across `pointermove` deltas, decays 0.85× per step when idle. **Buttons**: ⌖ Shake (random per-drop impulse 180–560 px/s with slight upward bias), ◌ Splash (push every drop outward from screen center, useful for clearing a stuck puddle), ⌬ Magnet toggle, ↺ Reset (rebuild the particle set). Welcome auto-shake fires 220 ms after load so the first frame isn't a sad puddle. **HUD**: top-left has a Major Mono Display "Liquid Mercury" title + Cormorant italic tagline + IBM Plex Mono meta row (drop count, gravity, impulse count). Top-right shows FPS, updated every 800 ms. Center has a fading "drag the silver — shake the room" hint in Cormorant italic; it fades out on first interaction. **Aesthetic**: pure dark-mode — `#04040a` void with three layered radial gradients (cool blue top-left, warm bottom-right, central dark). Silver `#d2d2dc` text, mist-grey accents, no neon colors anywhere. Vignette darkens corners; grain via `repeating-linear-gradient` at low alpha + overlay blend. Fonts: Major Mono Display + Cormorant Garamond italic + IBM Plex Mono. **WCAG basics**: stage `role="application"` with summary aria-label, every control has `aria-label`, Magnet button uses `aria-pressed`, ≥44 px button targets, focus-visible silver outline. Pollinations OG flux seed 80212.

## issues
- The `filter: blur(7px) contrast(24)` is GPU-accelerated in modern browsers but on older Android Chrome it can drop to 30 fps with 160 particles. The 96 default count is the sweet spot.
- Soft collisions are O(n²) — at the 160-particle max that's ~12.7k pair checks per frame. Still fast enough at 60 fps on a 2020 phone. If we ever want 400+ particles we'd switch to a spatial hash.
- The "magnet" mode can occasionally cause a tight cluster to oscillate (drops attracted to cursor while colliding outward). Switching the magnet off settles them.
