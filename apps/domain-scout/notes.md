# domain-scout

## log
- 2026-05-27: Initial build. Live tool that rolls 100 random 12-char `.com` domains per click and runs each through Cloudflare's DNS-over-HTTPS to classify it. **Generator**: `crypto.getRandomValues` (fallback `Math.random`) picks 12 chars from `[a-z0-9]`, retries up to 8 times to guarantee ≥1 letter AND ≥1 digit so we don't ship pure-numeric strings (mostly squatted) or all-letter ones (boring for this tool's framing). Dedupes via `state.seen[domain]` across rolls — duplicates are rerolled. **DNS check**: GET `https://cloudflare-dns.com/dns-query?name=<X>.com&type=NS` with `accept: application/dns-json` header, 6s `AbortController` timeout. **Classification**: `Status:3` NXDOMAIN → `available`; `Status:0 + Answer[]` with NS hostname matching `PARK_HOSTS` substring list → `parked` (+ surface which service in the label, e.g. "parked · sedo"); `Status:0 + Answer[]` other NS → `taken`; everything else → `available` (best guess). `PARK_HOSTS` covers ~25 well-known parking/for-sale services (sedo, parkingcrew, bodis, dan, undeveloped, afternic, uniregistry-dns, above, parklogic, parkingpanel, parkingpage, fabulous, voodoo, hugedomains, buydomains, brandbucket, namebrightdns, parkdns, parking-, cashparking, dnsowl, sedo.com, efty, whoisguard, parking.bodis, parkinghorse). Deliberately EXCLUDES `domaincontrol.com` (GoDaddy default) and `registrar-servers.com` (Namecheap default) — they're used by real active sites too, so matching on them caused too many false positives. **Concurrency**: 8 parallel checks via tiny `pumpQueue()` runner; ROLL_SIZE 100 means a roll dispatches 100 queued items, 8 in flight at any time, ~13s end-to-end for a full roll at typical DoH latency. **Card states + colors**: queued (silver dim), checking (cyan pulse), available (phosphor green w/ glow + tinted bg), parked (amber w/ tinted bg), taken (red, opacity 0.55 to fade them out), error (magenta). Each card: domain in JetBrains Mono w/ silver `.com` TLD suffix, status row with colored dot + label, copy-to-clipboard button. **HUD**: 4 status counter pills (avail/parked/taken/checking) + total in header, 5 filter chips in toolbar (all/available/parked/taken/checking) + clear, live `aria-live=polite` on counters. Roll/clear via `R`/`C` keyboard shortcuts (gated on non-input focus). **Memory bound**: `MAX_CARDS = 600`; on roll, oldest items are spliced + their seen-keys pruned to make room. **Defensive init**: try/catch on AbortController construction (older browsers), `window.error` listener → magenta toast, copy falls back to textarea+execCommand. **Aesthetic**: phosphor green on near-black with faint horizontal scanline overlay (`mix-blend-mode: overlay`, opacity 0.025), Major Mono Display title with phosphor-glowing `·` separator, VT323 cursor-blinking tag line, JetBrains Mono everywhere else. Tagline reads "12-char .com hunter · live DNS sweep · 100 per roll".

## issues
- **Heuristic, not authoritative.** NXDOMAIN from DNS only signals there are no NS records cached at the .com authoritative servers — it does NOT confirm the domain is actually available to register (could be reserved, in pendingDelete, in registrar hold, etc.). Footer explicitly warns this; users should always confirm with a registrar before purchase.
- **Parking detection misses generic-NS parking.** Domains parked on AWS Route53 / Cloudflare / generic registrar NS aren't flagged (e.g. `sex.com` is parked-for-sale on `awsdns-*` and would show as "taken"). Only services that use distinctive NS hostnames get matched. Acceptable — false-negative on "parked" just routes them to "taken" which is still a useful signal (not available).
- **Cloudflare DoH rate limits.** Free tier is generous (~100 req/min sustained) and our 8-parallel + 100/roll fits comfortably, but a user mashing the roll button could push past. No client-side throttle between rolls; rely on Cloudflare's own backpressure.
- **No cache.** Each roll generates fresh domains so a dedupe cache isn't useful across sessions. Within a session `state.seen` prevents the same 12-char string being rolled twice.
- **Filter "checking" includes "queued"** — the queued items haven't been dispatched yet but visually they live in the same bucket since both will resolve into a final state shortly.
