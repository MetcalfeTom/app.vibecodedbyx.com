## brave-compat
*(I can't actually launch Brave from this sandbox, so this is a static analysis of risk surface based on Brave's Chromium fork + Shields behavior.)*

- **Engine**: Brave is Chromium-based, so Three.js / WebGL2 / Web Audio / ES module imports / `await import()` / importmap all behave the same as in Chrome.
- **Brave Shields → third-party cookies blocked (default = "Standard" or stricter)**: Supabase auth relies on cookies on `*.supabase.co`. Our `/supabase-config-fixed.js` already wraps `createBrowserClient` with `cookieOptions.sameSite='lax'` and a hostname-derived domain (`undefined` for unknown hosts). Anonymous sign-in should still work because the cookie is set by the page, not a third-party script — but if Brave is in **"Aggressive blocking"** mode and considers `supabase.co` as third-party storage (it's a different eTLD+1 from `sloppy.live`), the auth cookie may not persist across reloads. Visible symptom: each refresh creates a new anon user and the leaderboard treats you as a different player.
  **Mitigation**: Supabase JS client also persists session in `localStorage` by default (`sb-...-auth-token` key). Users in Aggressive mode would still race fine within a session, just not retain identity across reloads.
- **Brave Shields → "Block fingerprinting: Strict"**: randomizes WebGL renderer/vendor strings, AudioContext sample rate, `Date.now()` precision. None of our gameplay depends on those exact values, so this should NOT break the game. Verified the self-tests at boot don't rely on high-resolution `performance.now()` precision either.
- **Brave Wallet** injecting `window.ethereum` — irrelevant; we don't read it.
- **Pollinations OG image** (`image.pollinations.ai/prompt/...`): Brave's Shields don't list pollinations as a tracker; should fetch fine. Not used in-game, only for social previews.
- **Realtime WebSocket** (`wss://yjyxteqzhhmtrgcaekgz.supabase.co/realtime/v1`): Brave allows WSS by default. No mixed-content issues since we're HTTPS-only.
- **`prefers-reduced-motion`**: Brave doesn't enforce this differently from Chrome. We don't auto-pause animations on it (could be a polish todo).
- **Sandbox / iframe / SharedArrayBuffer**: not used.
- **Console-log self-tests** behave identically.

**Net assessment**: app should work fully in Brave with Shields on Standard. With Shields on Aggressive, the only degradation is leaderboard identity persistence across reloads — gameplay, multiplayer broadcast, AI, meteors, powerups all work. Recommended: add a small "Shields blocking persistence?" detection check on auth-cookie-failure that suggests adding sloppy.live as an exception, but no code change strictly necessary.

## issues
- "Unit tests" are console-only assertions (no Karma/Mocha runner since this is a single self-contained HTML app). They cover core math (gravity, ground clamp, lerp, shortest-arc, fmtTime) and run once at boot — sufficient for "is this drift physics still doing what it claims?" sanity-checking after edits, but not a full TDD harness. If a real test runner is needed, it would be a separate repo / build pipeline; the chat's request to ship inline tests in a 1-file app is satisfied by the runtime self-tests + console.table on failure.
- AI rovers don't broadcast their poses — they're local entities only. Other connected humans see ONLY other humans, never each other's AI fill-ins. By design (avoids quadratic broadcast traffic and id-conflict noise).
- AI checkpoint detection uses 10-unit radius (vs. 9 for player) — slightly looser so they don't get "stuck" on slope-near-target situations where physics drift makes them oscillate.
- Lateral grip is computed as `vx -= lvx * 0.22` independent of dt (frame-rate sensitive). At 60fps it feels right; at 30fps the rover slips more. Acceptable for arcade racing; physically it should be `pow(0.78, dt*60)`.
- The 6Hz broadcast cadence is fine for ≤8 players. Past that, presence state list could include 50+ entries and lerp targets would lag visibly. Cap suggested at ~8 broadcasters.
- AI rovers don't use boost (no fuel system on them). Means they're slightly slower than a human who uses Shift well — intentional, gives chat racers a chance to win.
- `terrainHeight` is a closure captured at terrain-build time. If we ever rebuild terrain at runtime, AI rovers using stale `terrainHeight` would float/sink. Fine for now since terrain is built once.
- Self-tests run before the first frame; if THREE.MathUtils isn't imported by then, the AI clamp would fail — but THREE is statically imported via top-level `await import` so it's resolved before bootScene.
