# neon-cruise-zombies-3d

## log
- 2026-05-08: shipped — three.js port of the 2D `neon-cruise-zombies` game, **first-person**, with **WebXR/VR support**, **head-bob**, and a **creepy AI Director voice**. Two chat-ask trees converged on the same pivot (top-down 3D → first-person VR with director chatter); shipped as the consolidated build.
  - **Stack**: three.js r128 from cdnjs + `examples/js/webxr/VRButton.js` from jsDelivr. Single ~1500-line HTML, no build step.
  - **Deck system, weapons, enemies, AI director ported 1:1 from the 2D version.** Same five ASCII tile maps (Sundeck → Promenade → Atrium → Cabins → Engine), same five weapons (pistol / shotgun / smg / minigun with spin-up / grenade with arc + splash), same three enemy kinds + Captain Greaves boss, same enemy-collision via `tryMove`, same pickups, same Web Audio synth blips. Only the camera, input, and rendering layer changed.
  - **3D world build**: each tile = 1 world unit. Walls = `BoxGeometry(1, 1.6, 1)` with deck-accent emissive. Obstacles = smaller boxes. Pool tiles = thin transparent cyan slabs. Floor = single backing plane + per-tile checker. Stair-down = a glowing slab + 3 descending arrow bars + a co-located `PointLight` so it reads as a beacon from across the deck. Per-deck fog colour + a tinted `PointLight` near the centre.
  - **First-person camera**: `PerspectiveCamera` lives inside a `playerRig` `Group` that translates around the world. Yaw + pitch on `player.{yaw, pitch}`; mouse-look uses **pointer lock** for true FPS feel — click the canvas to lock, Esc to release. Sensitivity 0.0023 rad/px. Pitch clamped to ±(π/2 − 0.05) so you can't roll the camera upside down. WASD movement is **camera-relative** (W is "forward in the direction the camera looks"), with `Shift` sprinting at 1.45×.
  - **Crosshair** centred on viewport via two thin gold rects with neon shadow, only visible while pointer-locked. A "click to capture cursor · esc to release" prompt pulses while not locked.
  - **Head-bob** (chat ask): while moving, `player.bobT` advances at `8 × sprintMul` rad/s and drives both a vertical sine on the camera (`sin(2·t) × 0.045 × bobAmount`) and a **lateral cosine** sway (`× 0.025`). For the requested nausea: also a tiny **roll** wobble on the camera Z-axis (`× 0.018`) so the horizon doesn't sit flat. `bobAmount` eases toward `clamp(speed/5, 0..1)` at `dt × 8` so it ramps up/down smoothly. Bob is **disabled in XR** — VR users get a vetted ride to avoid the simulator-sickness lawsuit; the chat ask was for the desktop FP experience.
  - **WebXR / VR support** (chat ask): `renderer.xr.enabled = true`, `setReferenceSpaceType('local-floor')`, `THREE.VRButton` injected into the body (and re-styled via `[data-vr-injected]` selector to match the synthwave aesthetic — magenta border, Press Start 2P font). Game uses `renderer.setAnimationLoop(loop)` (three.js's XR-aware ticker) instead of `requestAnimationFrame`. Two controllers wired via `renderer.xr.getController(0/1)` — each gets a small glowing cone tip for visual presence, and their `selectstart` / `selectend` events map to fire-down / fire-up. Locomotion in VR reads thumbstick axes from controller `gamepad.axes`: left thumbstick = strafe + forward (relative to head yaw), right thumbstick X = **30° snap-turn** when crossed past 0.7 magnitude (gated to one snap per cross by `player._snapped`).
  - **Aim direction**: in flatscreen, `getAimXZ()` reads `camera.getWorldDirection()` and projects onto the XZ plane → angle. In VR the same call works because the headset has already written the camera transform. Bullets fire from the player's foot position (`player.x, 0.9, player.z`) along the aim angle, so where you LOOK is where you SHOOT — no separate reticle required.
  - **Viewmodel**: a small floating chip/box pinned to `camera.position (0.22, -0.20, -0.42)` retinted to the current weapon's color. Per-shot recoil twitches the model 60ms toward the player. In VR the chip floats at the same eye-relative location so it doesn't drift in your peripheral vision; the visible weapon is the controller cone tip that follows your hand naturally.
  - **Creepy AI Director voice** (chat ask): the existing intensity-tier system (LULL / BUILD / PEAK) now triggers a **director announcement** on every tier transition. `CREEPY_LINES` per-tier bank: LULL = ambient-haunting ("they hear your footsteps", "the buffet was never restocked", "someone is humming below decks"); BUILD = approaching dread ("the captain knows you are aboard", "a tour group is coming", "the ice machine has stopped working"); PEAK = arrival ("the director is laughing", "they have your scent now", "guest services is closed", "you should not have boarded"). Picked at random from the tier bank, displayed as italic Audiowide text fading in at the top-third of the viewport for 2.4s, tinted by tier (lilac LULL / gold BUILD / pink PEAK) with neon glow shadows. Director also spawns physical pressure waves of zombies in LULL (mostly walkers from offscreen) and bumps in BUILD (runners + bloaters), throttling in PEAK.
  - **Enemy 3D models**: walkers / runners / bloaters each get a deterministic `_decor` tourist outfit baked once per spawn — body cylinder + Hawaiian-shirt band ring (one of 6 palettes) + 3 horizontal pattern stripes + sun-hat brim disc + sun-hat dome cap + colored neon torus band + glowing red eyes. Captain Greaves: bigger 1.4-tall body + bloated `SphereGeometry` belly + navy uniform jacket cylinder + gold trim torus + 3 gold spherical buttons + gold epaulets + peaked cap with brim/dome/band/scrambled-egg cluster, plus a co-located red `PointLight` aura. All material `MeshStandardMaterial` with `emissiveIntensity` boosted on hit (the per-frame `e.flash` value drives the body's emissive intensity). Death spawns 12 directional particles + a 5-particle blood pool that lingers 4s.
  - **Bullets / particles in 3D**: bullets = small spheres with a basic material in the weapon color. 3-step sub-stepping prevents tunnelling. Hits flash the enemy emissive + spawn 5 hit-puff particles. Grenade = arcing sphere with `arcY = 0.4 + sin(prog·π) × 1.4`, tumbles via `rotation.x/y` on each frame, bounces off walls with 0.6 elasticity, on detonation calls `explodeAt` (same falloff math as 2D) + spawns a `RingGeometry` shockwave that expands by rebuilding its geometry per-tick. Particles are tiny spheres with transparent materials that fade alpha over their lifetime; gentle gravity (-4 m/s² on Y) for non-blood, blood pools lie flat.
  - **Lighting**: `AmbientLight` 0.18 + `DirectionalLight` (sun) 0.42 from above-front + a **player point light** that follows the player at head-height in cyan + a per-deck tint point light at scene centre.
  - **Mobile**: virtual joystick still works — moves the player. Look on touch is auto-aligned to movement direction (no pointer-lock equivalent on touch). Fire/dash buttons unchanged.
  - **2D fallback link**: small magenta link in the bottom-right corner pointing back to `/neon-cruise-zombies/` for users who prefer the original.
  - **Accessibility**: rem units, semantic `<canvas role="application">`, skip link, `aria-live="polite"` on HUD pills + director-msg, focus-visible white outlines on buttons, ≥2.6rem touch targets, `prefers-reduced-motion` no-ops the bob roll easing animation + flashing HP-crit + deck banner anim.

- 2026-05-08: **WASD + dash vector-math fix** — chat (latino_supreme_) reported "forward moves left", a clear 90° rotation offset. Three concurrent issues:
  1. **W/S inverted**: original code used `fwd -= 1` for W (treating fwd as a "into the screen" direction with -Z forward), then a formula that resolved to `mz = strafe*sinY - fwd*cosY`. With yaw=0 + W → mz=`+cos(0)`=`+1`, i.e. moves toward `+Z` (backward). Aligned the convention: `W → fwd += 1` (positive = forward), `D → strafe += 1` (positive = right), formula now `mx = strafe·cosY - fwd·sinY; mz = -strafe·sinY - fwd·cosY` which matches the three.js camera-forward basis `(-sin(yaw), -cos(yaw))` and right basis `(cos(yaw), -sin(yaw))`. Verified analytically + via `node` sanity check (W at yaw=0 → mx=0 mz=-1 ✓; W at yaw=π/2 → mx=-1 mz=0 ✓).
  2. **Joystick Y-axis sign**: joystick `dy` is positive when pulled down on screen. Original mapping `fwd += joy.dy` made down-on-stick = forward in the new convention; flipped to `fwd -= joy.dy` so down = backward, consistent with VR thumbstick conventions.
  3. **VR thumbstick Y**: same flip — left controller `axes[3]` (typically Y) is positive when stick is down; was `+=`, now `-=` so headset down-on-stick = backward.
  4. **Dash fallback**: when standing still, the dash used `cos(player.facing) / sin(player.facing)` as a math-plane angle, but `player.facing` was set to `lookYaw` (the camera's Y-rotation), giving a 90° offset (yaw=0 → dash direction `(1, 0)` = `+X` instead of `(0, -1)` = `-Z`). Now uses `(-sin(player.yaw), -cos(player.yaw))` directly — same camera-forward formula.
- 2026-05-08: **EMBARK fix** — title overlay wasn't dismissing on click. Root cause: `loadDeck(idx)` still ran `player.mesh.position.set(...)` from the original top-down draft, but the FP refactor turned `buildPlayerMesh()` into a no-op (the camera lives in `playerRig` instead of a body mesh). `player.mesh` was therefore `null`, the line threw a TypeError, `startGame` aborted before reaching the overlay-hide line, and the same throw at boot meant the rest of the wiring never ran on first load either — the EMBARK button looked dead because the click handler was attached but the consequent crash silenced everything. Fix: removed the dead `player.mesh.position.set` line in `loadDeck` and replaced it with a defensive `playerRig.position.set(player.x, 0, player.z)` so the rig snaps to spawn coords on each deck load. Also explicitly show the cursor-capture prompt + try `cvs.requestPointerLock()` from the same EMBARK gesture so the player drops straight into FP look without needing a second click (browsers that decline the auto-lock fall back to the visible prompt instructing them to click).

## issues
- The diagonal grass-stripe overlay from the 2D version isn't ported — three.js floor uses checker tile shading instead. Acceptable trade-off for 3D.
- Pointer lock can't be force-engaged before the user gestures; users have to click after EMBARK. Documented in the lock prompt.
- WebXR support depends on the user's browser + device. On non-XR devices the VR Button injects but reads "VR NOT SUPPORTED" — three.js handles this gracefully.
- VR-mode head-bob is off by design (chat ask was for the desktop nausea sensation, not actual VR sickness in users). If chat wants it on, it's a 2-line toggle.
- Captain boss aim/track is straight-line `tryMove` like the regular zombies — no telegraphed lunge. Same as 2D version.
- The grenade in FP fires its full configured range instead of "to the cursor distance" because there's no cursor on the ground. Range is a fixed 11 units. Could improve by raycasting where the camera is looking and clamping flight time, but full-range is more honest in FP.
- No reload/clip system carried over — same as 2D, ammo is a flat counter.
- Blackjack terminal is currently inert in 3D — terminal isn't recognised by `loadDeck` (atrium 'T' tile gets ignored). Future work: open the existing 2D blackjack overlay when the player walks within range.
