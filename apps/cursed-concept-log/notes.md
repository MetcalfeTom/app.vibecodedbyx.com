# cursed-concept-log

## log
- 2026-04-28: Created. **Database-backed forbidden-archive grimoire for chaotic app ideas** (merges two simultaneous chat requests: "dedicated cursed concept log with a database to archive our worst ideas" + "database-backed Cursed Concept Log where viewers can submit and upvote their most chaotic app ideas"). **Two-table schema** (via `mcp__supabase__create_table`): `cursed_concepts` = `title` (≤80), `description` (≤280), `category` (text key), `curse_level` (int 1-10), `submitter` (≤20) + auto `id`/`user_id`/`created_at`/`updated_at`, indexes on `created_at` and `curse_level`; `cursed_concept_votes` = `concept_id` (bigint FK reference, untyped) + auto cols, index on `concept_id`. RLS default: read-all, insert/update/delete own rows by `user_id = auth.uid()`. **Why two tables**: counter column on concepts wouldn't allow non-owner endorsements (RLS only lets row owner update). Separate votes table = any authed user can insert/delete their own vote row, client aggregates count via `voteIndex` Map keyed by concept_id. **Auth**: anon via `supabaseSession()` from `/supabase-config-fixed.js` ESM. **Realtime**: channel `cursed-concept-log-v1` listens to `postgres_changes` INSERT on `cursed_concepts` (skips dupes already in state by id), INSERT on `cursed_concept_votes` (skips own user_id to avoid double-count after optimistic update), DELETE on `cursed_concept_votes` (same self-filter). **8 categories**: tech (📱), game (🎮), art (🎨), food (🍔), social (👥), commerce (💰), occult (🔮), misc (❓), each with a category-specific badge color. **3 sort modes**: cursed (curse_level desc → votes desc → newest, default) / voted (votes desc → curse_level desc) / new (created_at desc). 'all' filter or per-category. **Curse level 1-10** mapped to **Roman numerals** via `ROMAN = ['','I'…'X']` shown in the wax seal; LVL_DESC array gives flavor labels (mildly off → unspeakable). ## issues
- **No edit-own-concept UI**: once submitted, can't fix typos in title/description. RLS allows update of own rows but no UI exposes this. Acceptable for "cursed" theme but worth a future pass.
- **No moderation**: any chat viewer can submit unhinged content. RLS only allows author-delete, no admin path. If griefing escalates, would need a soft-hide flag column + moderator UI.
- **Vote counts client-aggregated**: every load fetches all rows from `cursed_concept_votes`. Fine at <10k votes, but past that should switch to a `count()` rollup or postgres view. Watch list size as it grows.
- **Auto-seed race**: if two simultaneous first-viewers both see empty archive, both attempt seed insert → 6 duplicates instead of 3. Mitigated by the fact that one of them will see the realtime broadcast of the other's seed before its own insert returns. Edge case but possible.
- **Self-broadcast filter relies on user_id match**: works for INSERT (we own the row); for vote DELETE we also check `state.user.id === payload.old.user_id`. If a user votes from two browsers simultaneously, the second broadcast will be filtered correctly because both share the same anon user_id only within a session — different sessions = different ids.
- **Curse level slider visual gradient is purely decorative**: the actual `curse_level` int is the only stored signal. The gradient just helps the user feel the escalation while dragging.
- **No rate limit on submit/vote**: client-side only via the Inscribe button disabling during request. Anon spammer with direct API access could flood. Acceptable for now; would need postgres trigger for real defense.
