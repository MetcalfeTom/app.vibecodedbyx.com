↗ why direct links, not embedded waterfalls?
We checked. Four overlapping reasons make in-browser iframe embedding unreliable for almost all public SDR nodes:
  • mixed content — most receivers serve plain http://; this page is https://; browsers hard-block insecure iframes regardless of any other headers
  • X-Frame-Options / CSP frame-ancestors — some operators explicitly deny third-party framing
  • WebSocket scheme mismatch — the waterfall + audio stream over ws:// or wss://; an http-only receiver's ws:// can't be opened from an https page
  • third-party cookies — KiwiSDR sets a per-session cookie that modern browsers drop in iframe contexts (SameSite=Lax default)
A few HTTPS-native OpenWebRX+ instances might embed if their operator hasn't set frame-ancestors. Each card below now carries an 🔒 https or ⚠ http badge so you can spot the candidates — but the safe default is the deep link.
