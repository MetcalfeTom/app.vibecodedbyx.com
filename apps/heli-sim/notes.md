## log
- 2026-05-14: v2 — **3D camera tilt + water bucket + megaphone + 3D buildings via OSM**. Chat asked: "switch to a 3D view if possible and definitely add the megaphone and water bucket mechanics" followed by "investigate using Three.js or a similar library to simulate 3D buildings, and definitely add those water buckets and megaphone features." Answer to the Three.js part: **not needed** — MapLibre GL JS natively supports 3D building rendering via `fill-extrusion` layers, which is what's actually used here (Three.js would mean rebuilding the camera-sync, pickability, and tile texturing that MapLibre already provides). **Engine swap**: replaced Leaflet with MapLibre GL JS 4.7.1 (CDN, same Esri/OSM/CartoDB raster tile sources, still no API key required). Map is `interactive: false` so the sim controls camera. Each frame `map.jumpTo({ center: [lon, lat], zoom: lerp(z, targetZoom(alt), 0.08), bearing: heading°, pitch })` syncs the world to the heli — bearing = heading so the helicopter's forward is always "up the screen," and pitch eases between 40° (low/slow) and 70° (high/fast) for a real 3D camera feel without the player having to think about it. Tile layer toggle still works; re-creating the style preserves the buildings layer via a `styledata` listener. **Water bucket**: a CSS-rendered orange bucket on a 30-px cable hangs below the centered helicopter sprite. Toggle deploys it (`.show` opacity transition); sways with yaw input and a slow sinusoidal idle; water level visualized via a `--water-top` CSS variable that animates between 6 px (full) and 30 px (empty). Pressing **B** (or the rose Drop button) when deployed + full triggers `dropWater()` which (a) spawns ~30 falling-drop particles drawn on a top-layer FX canvas — animated from the sprite's center towards the heli's ground position in screen space with a slight gravity arc, (b) plants a "splash" record at the heli's lat/lon that expands as a cyan ripple ring + faint inner ring on the FX canvas (using `map.project(lonLat)` to track screen pos through camera moves), (c) zeros bucket water and starts a 4-second auto-refill timer (the bucket level animates back to full), (d) hit-tests against the fires array — any fire within ~85 m of the splash gets `extinguishing = 1.0` and is removed 600 ms later, bumping an "extinguished" counter. **Fires** (for bucket to chase): generated by **F** key or "🔥 spark fire" button — spawns a fire ~60-280 m around the heli at a random angle. Fires render on the FX canvas as a flickering radial-gradient halo + bright yellow core + 4 orbiting tongue particles, with their lat/lon projected onto the screen each frame so the camera tilt feels coherent. Extinguishing fires emit a steam puff before disappearing. Auto-spark of 3 demo fires on map.load so chat has targets immediately. **Megaphone**: **G** key or the jade "📣 shout" button opens a modal with a 220-char textarea. Submitting closes the modal, displays a yellow speech-bubble overlay anchored above the helicopter (`.shout-bubble` with `border-radius: 6px 18px 6px 18px` + a `::after` triangle tail), AND speaks the text via `speechSynthesis.speak(new SpeechSynthesisUtterance(text))` with pitch 0.72 / rate 0.95 / a male English voice if available — gives a low, slowed PA-system effect. Bubble auto-hides after `length·60ms + 1800ms` so longer shouts stay visible longer. First-pointerdown listener primes the speech voices list since some browsers lazy-init it. **3D buildings (OSM Overpass)**: toggleable "🏙 buildings" button. When enabled, queries `overpass-api.de/api/interpreter` (free, public, no key) with a `[out:json][timeout:8]; (way["building"](south,west,north,east)); out geom;` request inside a ~0.005°-radius bbox around the heli (≈500 m). The returned `way` elements with `geometry` arrays are converted to GeoJSON polygons, closed if necessary, and assigned heights — from `tags.height` directly, falling back to `tags["building:levels"] · 3.2 m`, falling back to 6 m, clamped 2-450 m. The GeoJSON is pushed into a `buildings` source and rendered by a `fill-extrusion` layer with a height-graded color ramp (`#3a4252` at ground → `#8aa4cc` for 200 m+ towers), `vertical-gradient: true` for soft side-shading, and 0.78 opacity. Refetched whenever the heli has moved > 150 m from the last fetch position (haversine distance). Hidden when toggled off; preserved across tile-layer style swaps via a `styledata` rebuild. So when you fly over Manhattan with buildings on, you actually see the actual buildings rise out of the satellite image, with their real heights.
- 2026-05-14: v1 — answer to chat: "is it possible to create a simple helicopter flight sim using google maps or an open source map api?" → yes, easily, with **Leaflet + free tile providers, no API key**. Built it. **Map stack**: Leaflet 1.9.4 from unpkg, three switchable tile layers (no key for any): Esri World Imagery (satellite, default), OpenStreetMap (streets), CartoDB Dark Matter (noir). Layer toggle in the dock cycles between the three; attribution string updates accordingly (legally required). All native Leaflet UI disabled (`zoomControl`, `attributionControl`, dragging, scroll-wheel zoom, double-click, keyboard, inertia, anim) so the sim has total control of pan/zoom. **Flight model**: position is `(lat, lon)`, heading in radians where 0 = north, throttle 0–1, target speed = throttle × `MAX_SPEED 78 m/s` (≈ 280 km/h ≈ 152 kt — a fast civilian heli). Per frame: speed eases toward target at `ACCEL 22 m/s²` or `DECEL 16 m/s²`; yaw input applies `TURN_RATE 1.4 rad/s`; altitude input applies `ALT_RATE 14 m/s` clamped 5–4600 m. Position step: `dLat = v·cos(h)/111320 m·s⁻¹·deg`, `dLon = v·sin(h)/(111320·cos(lat)) m·s⁻¹·deg` — i.e. true great-circle approximation valid below polar regions; lat clamped to ±85° to avoid the Mercator singularity. **Altitude→zoom mapping**: discrete steps (5m→z19, 80m→z18, …, 3500m→z13, +→z12) with a 5%-per-frame ease so zoom changes feel like physical climb/dive instead of a snap. **Helicopter sprite**: inline SVG with body, cockpit glass (radial-gradient shine), tail boom (amber stripe), skids, nav lights (red port, cyan starboard), tail rotor. Rotates with heading via CSS transform. Two `<div class="blades">` siblings at 0° and 90° spin independently — `rotorAngle += (4 + throttle·22) · dt`, so the disc spins faster as you throttle up. A faint radial blur disc sits behind them for the spinning-rotor downwash glow. **Shadow**: separate fixed-position SVG silhouette of the heli, translated diagonally off-center and scaled up as altitude rises (suggests vertical separation), with opacity fading at altitude. Body also bobs sinusoidally on a 250-ms cycle for liveliness. **HUD**: top-left stack — Major Mono Display title, eyebrow tag, four `.stat` cards (speed in knots, altitude in feet, heading °, lat/lon decimal). Top-right circular compass with N/E/S/W marks and a rose-red glowing needle that rotates inverse to the heli heading. Bottom dock has tile-layer toggle, teleport modal launcher, trail recording toggle, help. Bottom-left attribution stays compliant with the active tile provider. **Teleport modal**: 8 hand-picked destinations (Vilnius, Manhattan, Tokyo Shibuya, Grand Canyon, Mt. Fuji, Sydney Harbour, Lisbon coast, Reykjavík) + a free-text `lat,lon` input. **Trail recording**: when active, samples the helicopter's track into a Leaflet polyline (amber 2-px) added to the map. Polyline auto-extends each frame the heli moves at least ~2 m. **Inputs**: keyboard — W/S or ↑/↓ throttle up/brake, A/D or Q/E or ←/→ yaw, Space climb, Shift descend, P pause. Touch — left half = yaw (proportional to x), right half = throttle (proportional to y), multi-touch supported (can yaw AND throttle simultaneously). **Aesthetic**: amber phosphor `#ffb45a` over the live map, with global scanlines (`repeating-linear-gradient 0 1px 3px`, multiply) + corner vignette overlays so the sim feels like a mission-control HUD on top of a satellite feed. Major Mono Display + IBM Plex Mono. Pollinations OG flux seed 41414.

## issues
- Leaflet's tile fetch can be rate-limited at very high zoom levels (z18/19) if a viewer flies low fast. The 5%-per-frame zoom smoothing keeps zoom changes from spamming requests, but flying at z19 over busy areas may still see brief tile lag. Esri imagery is well-cached so this is mostly an OSM/Cartos concern.
- "Altitude" is purely cosmetic — there's no elevation data, so the heli never collides with terrain. At z19 over urban areas the visible buildings give an illusion of low flight, but the model is genuinely 2D over a flat earth approximation.
- The 111320 m/° latitude conversion is good to ~0.5% globally; we don't account for the WGS84 ellipsoid. Fine for a sim, not for navigation.
