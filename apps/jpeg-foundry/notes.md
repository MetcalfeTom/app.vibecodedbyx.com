# jpeg-foundry

## log
- 2026-05-27: Initial build. Chat ask was "WASM-based JPEG encoder for PNG/GIF/AVIF inputs", quickly pivoted to "bare-metal JPEG encoder from scratch in vanilla JS, no external libraries, handling Huffman tables and DCT". Initial draft used `@jsquash/jpeg` (mozjpeg via WASM) — fully replaced with a **from-scratch JPEG encoder** following ITU-T T.81 (1992) + JFIF 1.02. **What the encoder does**: (1) RGBA → YCbCr conversion via the JFIF/T.871 coefficients (Y = 0.299R + 0.587G + 0.114B; Cb/Cr around 0), with a white-matte step for transparent pixels (since JPEG has no alpha channel — without matting, transparent regions encode as black, which is rarely what the user wants). (2) 4:2:0 chroma subsampling — each 8×8 Cb/Cr block averages a 2×2 grid of source samples. (3) Forward 8×8 DCT via separable 1D-DCT (Y = C·X·Cᵀ where C is the orthonormal DCT-II matrix). Not as fast as AAN but bulletproof and readable. (4) Quantization using ITU-T T.81 Annex K standard tables (K.1 luma, K.2 chroma), scaled per the standard formula: `s = quality<50 ? floor(5000/q) : 200 - q*2`, scaled values clamped to [1, 255]. (5) Zigzag scan into 64-element arrays. (6) Entropy coding: DC coefficient differentially encoded against the previous block of the same component, AC coefficients run-length encoded with `(runLen<<4)|magnitude` symbols, ZRL (0xF0) for runs ≥16 zeros, EOB (0x00) when the rest of the block is zero. (7) Huffman coding using the standard sample tables from T.81 Annex K.3.3 (DC luma, AC luma, DC chroma, AC chroma) — built into `{value → {code, len}}` lookups via the Figure C.2 procedure. (8) **JFIF container**: SOI (FFD8), APP0 with JFIF identifier + version 1.02, two DQT segments (luma+chroma, stored in zigzag order per spec), SOF0 with H=2/V=2 Y sampling + H=1/V=1 Cb/Cr (= 4:2:0), four DHT segments (one per Huffman table), SOS with three-component scan, then entropy-coded data with byte-stuffing (every 0xFF in the bitstream followed by 0x00 per T.81 §F.1.2.3), bit-padded with 1s to a byte boundary, EOI (FFD9). **Bit writer**: custom `BitWriter` class with Uint8Array auto-growth, 24-bit accumulator, byte-stuff baked into `writeBits`. **MCU loop**: encodes blocks in standard scan order — 4 Y blocks (TL, TR, BL, BR within the MCU) then 1 Cb then 1 Cr per MCU. Yields to the event loop every 4 MCU rows via `await setTimeout(0)` so multi-megapixel images don't block input handling. **Input intake**: drag-drop anywhere or file picker. Decode uses browser-native `createImageBitmap(file)` for PNG/GIF/AVIF/WebP/JPEG/BMP, falls back to `Image.decode()` if createImageBitmap rejects (e.g. older Firefox + AVIF). Resulting bitmap drawn to a 2D canvas + `getImageData()` to feed the encoder. **Max dimensions**: 8000px/side guard to prevent absurd memory blowups. **Batch**: up to 30 concurrent jobs in the grid, oldest evicted FIFO when new ones arrive past the cap. **UI**: industrial foundry aesthetic — slate/ember palette, hot-orange accents on the ember icon and download buttons, dotted grid overlay, Major Mono Display title with glowing `∴` separator. Each job card shows input thumb + output JPEG thumb side-by-side (with checkerboard alpha-transparency pattern behind), dims, output size, compression ratio (vs input bytes), download + lightbox-preview buttons, dismiss. Quality slider 1-100 with 4 preset chips (archival 92 / balanced 78 / web 60 / thumb 40). Status stages: queued → decoding → encoding (wasm — kept the label since users don't care about implementation, and "encoding (wasm)" reads as "engine doing work") → forged / failed. **Verified end-to-end**: synthetic 32×32 gradient encoded at q78 → 749 bytes, decoded back with ImageMagick, pixels round-trip within ±2 levels of input, ImageMagick correctly reads back Quality:78 and sRGB colorspace. Tested at 16×16, 17×13 (odd dims that don't align to MCU), and 256×192 gradient at q10/q50/q90 — file sizes scale correctly (q10:1560B → q90:4722B for the gradient), all outputs identified by `file` as "JPEG image data, JFIF standard 1.02, baseline, precision 8, NxN, components 3".

## issues
- **DCT performance**: direct O(64) separable DCT is ~4ms per 8×8 block on V8. A 1080p image (8160 MCUs × 6 blocks = ~49k blocks) ≈ 200ms encode, fine for interactive use. AAN (Arai-Agui-Nakajima) would cut that ~3× but adds ~50 lines and the cognitive overhead wasn't warranted for this scope. If batch-encoding 4K images becomes a bottleneck, swap fdct8x8 for AAN.
- **Single-thread on main**: encoder yields every 4 MCU rows but still ties up the main thread between yields. For huge images consider moving the encoder into a Web Worker — would also free up the UI to show a real progress bar instead of just the "encoding" badge.
- **No alpha handling beyond white-matte**: PNGs/AVIFs with transparency get composited against white before encoding. Could expose a matte-color picker (white/black/transparent-as-black) as a setting.
- **Standard Huffman tables only**: optimal Huffman tables (custom-built per image from coefficient histograms) would shrink output by ~5-10% but require a two-pass encode. Not implemented — uses Annex K samples verbatim.
- **No optimize_coding or progressive**: encoder is baseline sequential only. Progressive JPEG would require multiple scans and interleaved coefficient bands — significant rewrite. Trellis quantization (mozjpeg's signature trick) also not implemented.
- **Quality scale isn't perceptually identical to libjpeg/mozjpeg**: same q value will produce slightly different file sizes vs libjpeg because we use the bare K.1/K.2 tables, while libjpeg has additional rounding tweaks and mozjpeg uses different perceptual tables by default.
