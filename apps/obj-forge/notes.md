- 2026-05-19: v1.10 — **flatExtrude (and alphaMask) now produce a real 3D solid with thickness — bottom plane + side walls**. Stacked chat asks: "check why the flat extrude export in obj-forge appears flat or broken in Blender" + "if flatExtrude is on, ensure the mesh still has a non-zero thickness". Root cause confirmed: v1.9's flat extrude only emitted a TOP plane (the `pos.y = amp` quads). Blender (or any 3D viewer with backface culling) renders a zero-thickness sheet — looks flat, looks broken from below, useless for 3D printing.
  - **Refactored quad loop** to track `kept[i*qCols+j]` (which quads survived the silhouette mask) instead of just emitting top triangles inline.
  - **New `makeSolid = flatExtrude || alphaMask` path** doubles the position + UV buffers (N more vertices at the same xz with y=0), then emits:
    - **bottom face** · same quad indices on the lower vertex set, reversed winding so normals face -Y;
    - **side walls** · for each kept quad, walks its 4 edges; where the neighbour is missing (out of grid OR not kept) it emits a wall quad with winding chosen so the normal points OUTWARD from the silhouette. Verified the cross-product for all 4 sides (top edge → -Z normal, bottom edge → +Z, left → -X, right → +X).
  - **Verified by sim** · full 4×4 grid → 60 triangles (18 top + 18 bottom + 24 perimeter walls). 8×8 grid with a circular silhouette → 124 triangles (42 top + 42 bottom + 40 silhouette-border walls). Closed solid, exportable cleanly.
  - **Side effect** · the alphaMask-only case (silhouette on, flatExtrude off) ALSO becomes a proper solid now — varying top height, flat y=0 bottom, walls along the silhouette + the outer rectangle. PNG sprite extrudes correctly as a relief-on-base 3D object.
  - **Limit** · `computeVertexNormals()` runs on the indexed mesh, so the 90° corner between top and walls gets smooth-shaded. Looks soft. Easy follow-up: `geo = geo.toNonIndexed()` would give crisp corners at the cost of more vertices — not done yet.
- 2026-05-19: v1.9 — **Flat extrude toggle for the heightmap generator** per chat ask "in obj-forge, add a Flat Extrusion toggle for the heightmap generator to skip pixel-depth math". New boolean param on the heightmap shape labelled `Flat extrude · skip pixel depth` (renders as the existing toggle-button style). When ON, every kept vertex's y is clamped to `amp` instead of `lum * amp` — produces a uniform-depth slab in the shape of the silhouette (pairs naturally with the existing Silhouette only toggle for cookie-cutter / 3D-printed-sticker output).
  - **Refactor** · added a parallel `lums` Float32Array to the heightmap builder. Every pixel writes its post-invert luminance into `lums[]` alongside the mesh `pos.y`. That way:
    - silhouette mask's height-fallback strategy uses `lums[]` (the source signal) rather than `pos.y` — so the cut still works correctly when flatExtrude has clamped every kept vertex to amp;
    - mask-debug overlay snapshot stores `lums[i] * amp` instead of `pos[i*3+1]` — so the red overlay shows the source image's relief even when the actual mesh is flat;
    - alpha-strategy + invert + the rest of the pipeline are unaffected.
  - **Demo placeholder** also respects flatExtrude: picking heightmap shape, flipping flat-extrude on, with no image uploaded gives you a flat 3-unit-square slab at y=amp (with the silhouette-only toggle off, you get a single solid slab; with it on the ripple's higher half becomes the silhouette and gets cut to nothing, since the lum-cutoff fires on the dim half).
  - Verified by sim: lum ∈ {0, 0.25, 0.5, 0.75, 1} → all heights = 2 when flatExtrude=true.

- 2026-05-18: v1.7 — **silhouette mask debug + auto height-fallback for opaque images**. Chat: "debug why some PNGs only extrude as rectangles even with silhouette on, check for alpha threshold issues". Root cause was the user's PNGs being fully opaque (alpha=255 everywhere) — many "transparent looking" PNGs actually have a flat solid background that was being added/flattened during save. The silhouette mask was correctly doing nothing on those. Three fixes:
  - **Alpha distribution scan** · After getImageData, walk all pixels once to find `aMin`/`aMax`, count fully-opaque vs partial vs fully-transparent.   - **Height-based fallback** · If `aMax - aMin <= 10` (uniform alpha), the silhouette mask now switches to cutting on the COMPUTED HEIGHT instead — skips any quad whose 4 corners are all below `amp * 0.05`. This handles JPGs and flat PNGs with dark backgrounds correctly. Combined with the Invert toggle it also handles light-background images (invert the lum, then the bright bg becomes y=0, gets cut).
  - **Diagnostic toasts** · When silhouette is on, surface the strategy + outcome: "no alpha · cutting on darkness (toggle Invert if subject is light)" for the height-fallback path, "silhouette mask matched nothing · image is fully opaque" when neither alpha nor height variation exists. - 2026-05-18: v1.5 — **ASCII FBX exporter for Blender + Unreal**. Chat: "can we add a basic FBX exporter to OBJ Forge for easier importing into Unreal and Blender?". Yes — hand-rolled an ASCII FBX 7.4 writer rather than pulling Three.js examples/exporters. ASCII FBX is the simpler half of the format (the binary half is proprietary and nasty); both Blender (File → Import → FBX) and Unreal (Content Browser → Import) accept it directly.
  - **Layout strategy** · convert geometry to non-indexed via `geo.toNonIndexed()` so every triangle carries its own position/normal/UV. Lets us emit normals + UVs as `MappingInformationType: "ByPolygonVertex"` + `ReferenceInformationType: "Direct"` — the simplest valid form with no IndexToDirect lookup tables.
  - **PolygonVertexIndex encoding** · each triangle written as `[a, b, ~c]` where `~c` is JS bitwise NOT (= `-(c+1)`). FBX uses negative-last-index as the polygon-end marker — verified with single-tri test: vertices `[0,1,2]` → `0,1,-3`.
  - **Structure** · FBXHeaderExtension (with timestamp + Creator) → GlobalSettings (UpAxis=1=Y-up matching Three.js, Blender/Unreal will rotate on import) → Definitions (counts) → Objects (Geometry + Model blocks with Vertices, PolygonVertexIndex, LayerElementNormal, LayerElementUV, Layer) → Connections (`C: "OO", geoId, modelId` + `C: "OO", modelId, 0` to attach to RootNode).
  - **`fbxPushArray(lines, indent, key, arr, isFloat)` helper** chunks 12 values per line with `,\n<indent>` continuation — produces readable + spec-compliant `Key: *N { a: ... }` blocks.
  - **UI** · panel export button is now a 2-button row (.obj orange primary + .fbx cyan alt). Viewport floating button is the same pair — both bottom-right of viewport. Keyboard: `e` = .obj, `f` = .fbx. `data-format` attribute on each button + single delegated listener so adding more formats later is one element + one entry.

- 2026-05-18: v1.3 — **heightmap extrusion convention fix**. Chat: "heightmap looks inverted or zero-offset, making models look like holes". Root cause: the v1.2 formula `(lum - 0.5) * amp * 2 * inv` mapped black to **-amp** and white to **+amp** (centred range [-amp, +amp]) — so any image that's mostly dark or mid-grey sat below the y=0 baseline, and from the default camera angle looked like a crater/hole in the ground plane. New formula `(invert ? (1 - lum) : lum) * amp` follows the standard displacement-map convention used by Blender's displace modifier, Unity's terrain, etc: black = ground (y=0), white = peak (y=amp), mesh always sits on or above the baseline. Verified by sim: lum=0→h=0, lum=0.5→h=0.6, lum=1→h=1.2. Invert toggle now flips the bright/dark interpretation cleanly while keeping the mesh above ground (so a "mountain" heightmap becomes an upside-down range of basins still sitting on the plane, not a mirror plunge below it). Default amplitude bumped 0.8→1.2 to compensate for the halved vertical range (old spread was 2×amp around zero, new spread is 1×amp upward) — visual impact stays similar. Max amp 2.5→3.0. Dropzone hint now reads "black = ground · white = peak" so the convention is documented in the UI.
- 2026-05-18: v1.2 — **heightmap pipeline rewrite for big files + broader format support + alpha handling + diagnostics**. Three chat asks converged on the same code path: (1) handle larger files + common formats; (2) debug pixel-processing stalls on certain formats; (3) PNG upload failing for deadbydaddyttv — likely the FileReader→dataURL stage choking. Replaced the whole loader.
  - **Fast decoder** · `decodeImage(file)` races `createImageBitmap(file, { imageOrientation: 'from-image' })` against a 15s timeout, with two graceful fallbacks: plain `createImageBitmap(file)` if the orientation option throws (older Safari), then a Blob-URL `<img>` element if even that fails. Avoids the FileReader→base64→`img.src=` round-trip entirely — base64 inflates a 25MB PNG to ~33MB and forces a second decode, which was the most likely cause of the deadbydaddyttv failure.
  - **Pre-downsample cache** · After decode, the source is downsampled once to a 1024-edge max canvas (`state.heightmapSource`) with a hard `MAX_INTERMEDIATE = 8192` ceiling (below the Chrome 16384, Firefox 11180, Safari 4096-or-16384 canvas-size limits). Every subsequent slider rebuild draws from this small cached canvas instead of re-sampling the original — so a 6000×4000 source still rebuilds in milliseconds when you drag the resolution slider.
  - **Defensive `getImageData`** · Both the downsample step and the per-rebuild sampling wrap `getImageData(0,0,1,1)` in try/catch with a placeholder-plane fallback + toast on tainted-canvas errors. `willReadFrequently: true` hint on both contexts so Chrome optimises for repeated CPU pixel reads.
  - **Format support** · accept attribute now includes png/jpeg/webp/gif/bmp/avif/svg+xml + image/*. Explicit upfront rejection for HEIC/HEIF/TIFF/RAW/PSD via `KNOWN_BAD_EXT` + `KNOWN_BAD_MIME` regexes with a helpful "re-save as PNG or JPG" toast — these would silently fail in `createImageBitmap` on Chrome/FF anyway.
  - **Alpha-aware luminance** · pixel conversion now composites RGB over black using the alpha channel (`r * (a/255)` then standard ITU-R BT.601 weights), so transparent PNG pixels read as low elevation instead of arbitrary subject-RGB.
  - **Size cap bumped** · 12 MB → 32 MB.
  - **Diagnostics** · 12 `console.info`/`console.warn` checkpoints across the load path so future bug reports show exactly which stage failed (file metadata, decoder choice, dimensions, downsample, taint check, thumb). Visible in DevTools console.
  - **Clear button now closes ImageBitmap** · explicitly calls `.close()` on the bitmap reference before nulling, releasing GPU memory immediately instead of waiting for GC.
  - Dropzone metadata line shows `OOOxOOO → CCCxCCC cached` when downsampled, just `OOOxOOO` otherwise. Bumped ~40→~48KB.

## issues
- Image upload stays in memory — bigger images at 256-res take ~50ms to rebuild on each slider drag (acceptable but noticeable).
- Heightmap normals from `computeVertexNormals()` look soft on grayscale gradients; consider edge-aware shading later.
- No texture coordinates emitted in OBJ; Blender will fall back to default UV. Add `vt` + `f a/t/n` triplets later if anyone wants textured exports.
- Smooth-shading flag (vs the current flat) would be nice as a toggle.
- File-size estimate is rough — actual OBJ tends to be smaller after decimal-stripping.
