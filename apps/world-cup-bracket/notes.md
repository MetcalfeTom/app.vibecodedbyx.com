# world-cup-bracket

## log
- 2026-05-07: shipped — World Cup 2026 knockout-bracket prediction app with a social group + leaderboard, neon-pitch aesthetic, and a consensus-based scoring system. Four chat asks bundled into one cohesive build.
  - **Bracket** is a 32-team knockout from Round of 32 → Final, rendered as a 6-column CSS-grid (R32/R16/QF/SF/F/Champion) with proper `grid-row` doubling per round so each match centres between its two upstream feeders. 31 match cells in total. Each match is two stacked button slots (`home`/`away`) — click a team to advance them, the loser fades + line-throughs, the winner glows lime + propagates to the matching slot in the next round. Replacing a winner upstream auto-clears any stale downstream picks via `clearDownstream(...)` so the bracket stays coherent.
  - **32 starting teams** are pre-seeded into a reasonable strongest-vs-weakest pattern (Argentina vs Saudi Arabia, France vs Austria, etc.). The actual qualifying/draw is still in flight as of 2026-05-07 — this is a prediction app, not a live tracker. Each team has flag emoji + 3-letter FIFA code + full name.
  - **Persistence**: `localStorage['world-cup-2026-bracket-v1']` stores the full rounds array; name/code/top-striker stored separately. URL hash `#group=<code>` populates the code field on load (so `sloppy.live/world-cup-bracket#group=neon-wolves-42` is the share link).
  - **Supabase backend**: new `world_cup_predictions` table (display_name, group_code, bracket jsonb, champion text, plus auto user_id/created_at/updated_at + default RLS). Anon auth via `supabase-config-fixed.js`. Save flow: read existing row by user_id, update or insert one-row-per-user. Bracket payload = condensed `{ v:1, r: [[...],[...],[...],[...],[...]], striker }` with just the winner per match position (24 slots filled when complete) — keeps rows tiny.
  - **Group invite system** (chat ask): every save attaches the user's row to a `group_code`. Default code is **auto-generated on load** from a 30-adjective × 30-noun word list with a 2-digit suffix → ~90,000 unique codes (`neon-wolves-42`, `hat-trick-falcons-87`, `electric-dragons-15`). Re-roll button generates a new one. Two copy buttons: `▼ copy code` (just the code) and `▼ copy link` (full `sloppy.live/world-cup-bracket#group=<code>` URL). The invite card has its own magenta-bordered glassmorphic styling so it reads as the primary social affordance.
  - **Top Striker pick** (chat ask): free-text input "Predicted Golden Boot" next to the display name. Stored inside the bracket JSONB as `bracket.striker`. Free-form so users can write any name (Mbappé, Haaland, Vinicius, an unknown Norwegian winger, etc.) without bias. Counts toward scoring as +5 if it case-insensitive matches the group's most-voted striker.
  - **Consensus scoring system** (chat ask): not "right vs the actual results" (the tournament hasn't happened) but "agreement with the wisdom of your group". `computeConsensus` tallies every match's most-voted winner across all members. `scoreRow` awards round-weighted points (R32:1 · R16:2 · QF:3 · SF:5 · F:8) for matching the consensus pick at each match, +5 champion bonus for matching the group's consensus champion, +5 striker bonus. Max possible is 16+16+12+10+8+5+5 = 72 points → normalised to a 0–100 % score. Score displayed in a coloured pill on each member card (lime ≥50% · gold 25–49% · magenta <25%).
  - **Leaderboard** ranks members by score descending with rank numbers + medal-tinted top 3 (gold/silver/bronze rank glyph). Includes a special "▲ Group Consensus" card pinned to the top showing the group's consensus champion (with vote count `7/12 picks`) + consensus striker — only appears once ≥2 members have saved.
  - **Neon-pitch aesthetic** (chat ask): full-page fixed SVG of a soccer field as background — outer touchline, halfway line, centre circle + spot, both penalty boxes with goal areas + penalty arcs, four corner arcs. All strokes in cyan `#7af0ff` with a 0.25rem drop-shadow for the neon glow, opacity 0.18 so it doesn't fight the bracket. Mowed-grass stripes (alternating 84px-wide bands of cyan + lime tinting) layer on top via repeating-linear-gradient, masked by a centre-radial fade so the edges drop off. Body bg is a vertical pitch-green gradient (`#061d10 → #03110a`) plus radial cyan + magenta corner glows.
  - **Typography**: `Big Shoulders Display` weight 900 for the title (W:HITE with cyan/lime glow shadows, "26" in lime + magenta accents) and 700 for round headers + member names; `Bowlby One SC` for the predicted-champion banner; `IBM Plex Mono` for everything else. Title eyebrow reads "USA · Canada · Mexico · June – July" in cyan 0.34em-tracked all-caps.
  - **Layout**: 2-column desktop (bracket area + 18rem sticky social panel); below 920px the social panel drops below the bracket and the bracket grid auto-scrolls horizontally so the 6 columns stay readable on phones.
  - **Polish**: tab strip above the bracket auto-marks the round you've completed (active `active` class moves R32 → R16 → QF → SF → Final → Champion as you fill them out). Toast notifications at the bottom for save/copy/error states. Connection pill in the header (lime when Supabase auth is live, dim when offline-only).
  - **Accessibility**: rem units, semantic `<header>`/`<main>`/`<aside>`/`<section>`, skip link, `<button>` slots with `aria-label="Pick Argentina"`, `aria-live="polite"` on the toast, focus-visible outlines, ≥2.4rem touch targets, `prefers-reduced-motion` no-ops the connection pulse.

## issues
- The 32 pre-seeded teams are educated guesses for World Cup 2026 (qualifying isn't fully resolved as of 2026-05-07 — Italy in particular is a question mark). Acceptable for a prediction app where users can pick whoever they want; users can re-imagine the bracket either way by clicking through.
- Consensus scoring is a self-referential metric (you score against the group's average, not actual outcomes). When the real tournament starts, a "real-results" mode would need a separate column and an admin-only edit path. Out of scope for now.
- The bracket grid uses fixed 6 columns. Below ~480px screens, the cells get tight — horizontal scroll engages but tap targets shrink. A "vertical-stack" mobile variant would be a future polish.
- One-row-per-user means a user can only join one group at a time. Switching codes overwrites their previous row's group_code. Acceptable for an MVP.
- Empty groups (only the user themselves) show their pill as 0% because there's no consensus to match. The "Group Consensus" pinned card hides until ≥2 members. Not a bug, but might confuse a solo user — toast on save suggests "invite friends to see scores".
