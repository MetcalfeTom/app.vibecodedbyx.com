# sloppy-ops

## log
- 2026-04-30: Created. **Cluster ops dashboard with real filesystem + git data + live mock metrics.** 4-panel layout: top totals row (SERVICES / DB POOL / DEPLOYS-24H / REQ/S), STORAGE bars (left, 2-row span), DB CONNECTIONS sparkline + numerics (top-right), RECENT DEPLOYMENTS list (bottom-right). Each panel is a glassy slab with a subtle inner border bracket and 28px inset cyan glow. **Aesthetic**: `JetBrains Mono` everywhere, `Major Mono Display` for big numbers, faint 32px CSS background grid behind everything, 1px screen-blend scanlines. Sysadmin terminal vibe — green/cyan/amber/hot palette. Top bar shows region, uptime, and a live UTC clock that ticks every second. Footer is a `$ tail -f` console line with a blinking caret. **Storage panel**: each service row is `name | gradient bar | used / alloc unit`; bars run green→cyan→amber→hot as fill % crosses 70%/88%. Total at the top shows `used / alloc · pct%`. **DB panel**: huge `Major Mono Display` count (e.g. "78") with delta indicator (+/- vs last sample), 60-sample sparkline rendered to a canvas (path stroke + radial-gradient fill underneath + glowing dot at the head), plus 4 sub-stats (IDLE / WAITING / P95 LATENCY / QPS) refreshed every 1.5s. Sparkline uses `60 + sin(t*0.04)*18 + noise` so it gently wanders rather than spasms. **Deployments panel**: scrollable list with status pill (✓ OK / ✕ FAIL / ⚡ LIVE), app name, `<code>hash</code>` + commit subject, relative time. Live status has a pulsing amber animation. **Real data layer**: a `data.json` is generated at deploy time by a Python build script (`du -sb apps/*/` for sizes + `git log --pretty=...` parsing for the last 30 commits touching `apps/`). The dashboard `fetch('./data.json')` on boot — if it loads, **SERVICES** are replaced with the real top-30 apps by size (bytes → MB) and **deploys** are replaced with real commit hashes + ISO timestamps + commit subjects. Fallback to baked-in mock arrays if the JSON is missing or fails to parse. Storage allocation tier auto-picks from `[1, 5, 10, 25, 50, 100, 250, 500] MB` as the first tier above `usedMB * 1.4` so each bar reads at ~60-70% fill (the realistic SRE "almost full" feel). Footer flips to `snapshot · <generated_iso>` when real data is loaded so users know how fresh it is. **Mutation loop**: storage values wiggle by ±0.06 MB every 2.2s with occasional drift sign flip, the sparkline samples every 1.5s, deployments re-render every 3s (so relative times tick up), totals every 1.5s. Every 18s a mock fresh deployment may prepend to the list (only when running on mock data; real data list is static). Mobile responsive (`@media max-width:540`) collapses totals to 2 columns. Pollinations OG.

## issues
- **Real data is static at deploy time.** Browser JS can't query the server filesystem live, so the `data.json` snapshot is a fresh-at-deploy-time picture; the dashboard timestamps the snapshot in the footer. Subsequent commits won't appear until someone re-runs the snapshot script and pushes (an automated `pre-commit` or post-merge hook that regenerates `data.json` would keep it fresh — todo).
- The mutation loop wiggles `s.used` slightly on real data too, which is a tiny lie (real disk usage doesn't change while the page is open). Acceptable since the wobble is < 0.1 MB. To freeze: set `s.drift = 0` for real entries. Currently we set `drift = (Math.random() - 0.5) * 0.04`, very small.
- Sparkline data is 100% synthetic — there's no actual database connection metric to query. Keeping it as a "vibe" chart, clearly under "DB · CONNECTIONS" but acknowledged in the meta line "postgres · pgbouncer".
- Deployment list shows the commit subject as the second line; some commits have very long subjects (Moon Rover entries are 100+ chars), truncated at 80 with an ellipsis.
- Generation script would be cleaner as a Node build step, but Python is what was available; runs in <1s.
- Region is hardcoded `us-luna-1` (mock) → flips to `us-vibe-1` once real data loads (still mock — we have no actual region).
