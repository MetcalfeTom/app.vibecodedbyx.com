- 2026-04-30: **Adaptive snapshot polling + LIVE SYSLOG with multi-select severity filters.** **Polling refactor**: replaced the brain-dead `setInterval(refreshSnapshot, 5000)` with an adaptive `Poll` controller. (1) Uses `If-None-Match` ETag headers + `cache:'default'` so the browser handles conditional GETs natively — unchanged data.json returns 304 with no body, no re-render. (2) Computes a djb2 hash of the response body as a content fingerprint; if it matches the last hash, treats it as idle. (3) After every 3 idle hits, doubles the polling interval (5s → 10s → 20s → 30s capped). Any real change snaps it back to base. (4) `visibilitychange` listener pauses polling when the tab is hidden and forces an immediate refresh on resume. (5) Fail-on-fetch backs off harder (idleHits = max(3)). (6) Force-refresh path (manual ↻ button) sets `cache:'no-store'` to bypass any edge cache and flushes the etag/hash. Net effect: a quiet stream that used to download 250KB×12/min will burn 1 fetch every 30s with mostly-304 responses; an active stream still gets sub-5s data. **LIVE SYSLOG** — new ≡ SYSLOG sidebar tab with a ring-buffer Log store (cap 500) fed by every reactive source on the page: snapshot loads/refreshes (`snap`), polling state transitions (`poll`), Supabase realtime postgres_changes for deployments (`deploy`) and notes (`note` — INSERT/DELETE), and presence channel sync events (`presence` — `viewers N → M`). ## issues
- **Real data is static at deploy time.** Browser JS can't query the server filesystem live, so the `data.json` snapshot is a fresh-at-deploy-time picture; the dashboard timestamps the snapshot in the footer. Subsequent commits won't appear until someone re-runs the snapshot script and pushes (an automated `pre-commit` or post-merge hook that regenerates `data.json` would keep it fresh — todo).
- The mutation loop wiggles `s.used` slightly on real data too, which is a tiny lie (real disk usage doesn't change while the page is open). Acceptable since the wobble is < 0.1 MB. To freeze: set `s.drift = 0` for real entries. Currently we set `drift = (Math.random() - 0.5) * 0.04`, very small.
- Sparkline data is 100% synthetic — there's no actual database connection metric to query. Keeping it as a "vibe" chart, clearly under "DB · CONNECTIONS" but acknowledged in the meta line "postgres · pgbouncer".
- Deployment list shows the commit subject as the second line; some commits have very long subjects (Moon Rover entries are 100+ chars), truncated at 80 with an ellipsis.
- Generation script would be cleaner as a Node build step, but Python is what was available; runs in <1s.
- Region is hardcoded `us-luna-1` (mock) → flips to `us-vibe-1` once real data loads (still mock — we have no actual region).
