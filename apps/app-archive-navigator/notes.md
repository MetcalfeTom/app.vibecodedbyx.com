# app-archive-navigator · notes

## log
- 2026-05-13: v1 — searchable browser for all ~1,150 apps in /vibespace/apps. Chat request: "create an app called app-archive-navigator that lets users search and browse the eleven hundred apps by category and popularity" + "index the entire apps directory and create a searchable list." **Indexer** (`build-index.py`, run manually or via cron): scans every `apps/<slug>/index.html`, reads first 8 KB only, extracts `<title>`, `og:description` (or `meta name=description`), checks for `og:image`, records mtime + size. Heuristic categorizer assigns one of 10 buckets (game/sim/art/music/utility/chat/text/social/weird/misc) by keyword-pattern match on title+desc+slug. Skips `app-archive-navigator` / `sloppy-ops` / `sloppy-header` (infra). 1.15k apps indexed in <1 s, output `apps-index.json` is ~227 KB. **Front-end** (`index.html`): loads the JSON at boot, renders cards immediately, then loads `app_votes` from Supabase (anon read) and re-renders with vote counts merged. Sort options: popular (default — vote desc, mtime tiebreak), recent, oldest, alphabetical, shuffle. Category pills with per-cat counts; "all" pill shows total. Search input is space-AND across slug+title+desc (130 ms debounce). **Card design**: editorial card-catalog vibe — Newsreader italic title, JetBrains Mono category tag + slug + footer, IBM Plex Sans body, paper-cream `#f5e6cf` on warm-dark `#120c08` w/ amber `#d99547` accents + 7°/97° linen-grain backdrop. Cards link to `/<slug>/` in new tab. **Performance**: initial render 60 cards; IntersectionObserver sentinel near the bottom auto-loads +80 more (~600 px root-margin) so 1.1 k apps stay smooth. No virtual-DOM, just `innerHTML` batch replace on filter/sort changes. **Status pill**: idle → "loading index…" → "loading votes…" → "live · N apps with votes" (sage dot), or "votes unavailable · showing by date" fallback (rust dot) — if Supabase fetch fails, falls back to recent-first sort automatically. **WCAG basics**: semantic `<main>`/`<header>`/`<section>`/`<footer>`, `aria-pressed` on category pills, `aria-label` on controls, `:focus-visible` amber outline 2 px, 2.75 rem (44 px) min interactive targets, `prefers-reduced-motion` disables card hover transform. Mobile @640 px: single-column grid, controls stack. Pollinations OG flux seed 11157 (library card catalog warm amber).

## issues
- Vote aggregation pulls up to 50 000 `app_votes` rows client-side then tallies in JS. With ~1.15 k apps × low average vote count per app, this is well under that cap, but if votes ever grow into the hundreds-of-thousands range a server-side aggregate view (`app_vote_totals` materialized view) should be introduced to keep the page fast.
- Category heuristic is keyword-based and assigns ~31% of apps to `misc`. Improvements would need genuine semantic classification — could call Pollinations once per app and bake categories into the index, but that's an offline batch job not a runtime call.
- The index file is committed to the repo. It's regenerated by running `python3 build-index.py` (or attaching it to the hourly sloppy-ops cron). It is NOT auto-rebuilt on every visit — that would defeat the static-file performance benefit.
- Cards link to `/<slug>/` which works on sloppy.live (nginx routes each apps subdir as a site). On localhost the same URL pattern works because the dev server serves apps/ statically.
