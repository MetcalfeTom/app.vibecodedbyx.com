# metro-pulse

## log
- 2026-05-09: shipped — PWA metronome with high-precision Web-Audio scheduler, audio + Bluetooth latency calibration, and a beat-locked visual pulse (chat ask: "build a Bluetooth and audio synchronized metronome PWA with a high-precision clock and visual pulse").
  - **PWA**: `manifest.json` (standalone display, theme-color `#06070d`, inline-SVG 192/512 icons baked as `data:image/svg+xml` URLs so no extra fetches needed), `service-worker.js` registered on load (cache-first for the app shell, opportunistic cache for any same-origin GET, falls back to `index.html` offline). Apple-mobile-web-app meta tags + `apple-touch-icon` so iOS treats it as a home-screen app. `beforeinstallprompt` listener exposes the "↓ Install PWA" button on Chromium browsers; iOS users use the Share-sheet "Add to Home Screen" route.
  - **High-precision clock**: implements the canonical Chris Wilson "A Tale of Two Clocks" pattern — a 25ms `setInterval` scheduler tick wakes up, looks 100ms into the future against `audioContext.currentTime`, and pre-schedules every click within that window via `oscillator.start(time)` calls. So even when the JS timer jitters (which it does, especially on backgrounded tabs), the actual click fires at the sample-accurate scheduled time. The HUD's "Drift" readout shows JS-timer slack each tick, typically <2ms during normal operation, spikes when the tab goes background.
  - **Audio + Bluetooth latency calibration**: BT speakers/earbuds add 100-300ms of audio latency that desyncs the visual pulse from the audible click. Three layers of help:
    1. **Browser-reported latency** surfaced from `audioContext.outputLatency` (or `baseLatency` fallback) in the HUD's "Out-latency" pill — gives the user a baseline.
    2. **User offset slider** (-300ms to +300ms): shifts both the AUDIO scheduling time AND the visual-pulse anchor by the same amount, so light + sound stay in lockstep regardless of the BT chain. Negative offset = clicks fire earlier (compensating for late BT audio).
    3. **Tap-along calibration**: 8 metronome ticks at 1.5s spacing; user taps Space (or any key) on each click. App computes the median tap-vs-click delta across ≥3 valid taps, suggests the inverted offset, applies it automatically. Bounded to ±300ms.
  - **Web Bluetooth scan**: `navigator.bluetooth.requestDevice({ acceptAllDevices: true })` opens the OS picker. Web Bluetooth doesn't route audio (that's the OS's job, never the browser's), so the scan is purely informational — confirms a BT device is selected and surfaces its name + ID slug. Companion text reminds the user that BT inflates latency 100-300ms and points them at the slider/calibration. Graceful fallback for browsers without Web Bluetooth (iOS Safari, most Firefox builds): the slider + calibration still work fully.
  - **Visual pulse**: canvas 2D, requestAnimationFrame, renders against the same `audioContext.currentTime` as the audio scheduler. Each frame finds the scheduled click closest to "now", computes a triangular envelope `1 - |dt| / (beatDur*0.65)` clamped to [0, 1], and draws an outer radial-gradient halo + a pulse-width ring + per-beat indicator dots (lit on the active beat, magenta on accent, cyan otherwise) + a swinging gold pendulum line. Big translucent BPM watermark in the center of the ring as a constant readout. The visual envelope decays through the beat duration so fast tempos feel snappier and slow tempos feel calmer — same code path either way.
  - **Time signatures**: 2/4, 3/4, 4/4 (default), 5/4, 6/8, 7/8, 9/8, 12/8. **Subdivisions**: quarters, eighths, triplets, sixteenths. Accent first beat toggle. Changing meter or subdivision mid-run resets `currentBeat = 0` and re-anchors `nextNoteTime` so the accent always lands on beat 1 of the next bar.
  - **Click voices**: Sine (soft tone), Square (snappy click), Triangle (woodblock-ish), Cowbell (two detuned squares through a 1.4× bandpass at Q=5), Rim (bandpass-filtered noise burst at 3-4kHz). Accent = +625 Hz pitch + +0.21 gain.
  - **Tap tempo**: Space (when not focused on an input), or the dedicated `Tap` button. Resets if no tap within 2.5s. Averages the last up-to-6 intervals, clamps the resulting BPM to [30, 300]. Plays a rim-click on each tap so the user gets feedback even with the metronome stopped.
  - **Keyboard**: Space tap, P play/pause, ←/→ BPM ±1 (Shift = ±5).
  - **Tempo markings**: dynamic Italian marking under the BPM number (Grave / Largo / Adagio / Andante / Moderato / Allegro / Vivace / Presto / Prestissimo / Beyond presto).
  - **Aesthetic**: deep navy radial bg (`#122242 → #060710 → #02030a`), Audiowide for big numerics, Press Start 2P for tiny labels, JetBrains Mono body, Cormorant Garamond italic for tempo markings + brand tagline. Cyan/magenta/amber accent palette: cyan = beat indicators + ring outline, magenta = accent (downbeat), amber = pendulum + drift readout, acid = "ok" status. Single-column max-width 50rem layout — comfortable on phone, tablet, desktop.
  - **Accessibility**: `dir=auto`, `lang=en`, `<main>`/`<section>`/`<header>`, `aria-pressed` on the play-toggle, `aria-live="polite"` on the pulse meta status row, `aria-label`s on every range/select/checkbox, `:focus-visible` outlines, 2.75rem (44px) min interactive targets everywhere, skip link, `prefers-reduced-motion` removes toast transitions. The pulse canvas is purely decorative (`aria-hidden="true"`) — the live beat number announces via the `<b id="metaBeat">` inside the `aria-live` region instead.

## issues
- `audioContext.outputLatency` isn't implemented uniformly — Chrome desktop exposes it accurately, Firefox returns 0 or misses the property, Safari only fills it for some output devices. The slider + calibration are the durable path; the readout is an informational hint.
- Web Bluetooth has zero audio routing — it can pair to a BT device's GATT services but the OS still owns audio output. The scan UI is a UX nicety to remind the user about the BT-latency tradeoff, not a real "synchronization" mechanism. The actual sync mechanism is the user-offset slider and the tap calibration.
- Service worker caches `index.html` + `manifest.json` only — runtime additions (Google Fonts CSS) are network-passthrough and may not work fully offline first-load. The app shell still renders + plays clicks without fonts; just less pretty.
- Tempo presets are simple chips on the same panel. A "tap-to-set" preset that records the user's typical practice tempos would be a nice extension.
- No song/setlist mode yet — chat could ask for a queue of "30s at 80 BPM, then 30s at 120 BPM" later.

## design-notes
The "Bluetooth synchronization" framing in the chat ask is genuinely tricky — Web Bluetooth doesn't route audio, and once audio leaves the browser it's at the mercy of the OS audio stack. The realistic shipping interpretation is "calibrate the visual + audible click to match what the user actually hears through their BT speaker/earbuds." That's what this app does: surface the browser's reported output latency, expose a fine-grained user offset, and provide a tap-along auto-calibration that suggests the right offset based on the user's own perception.
