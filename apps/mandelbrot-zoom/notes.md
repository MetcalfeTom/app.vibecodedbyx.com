# mandelbrot-zoom · notes

## log
- 2026-05-13: **v1.1 — Server Mode toggle + simulated tile-farm cluster**. Per chat: "investigate if the mandelbrot-zoom can use a simple backend worker to pre-render tiles" + follow-up "build a Server Mode toggle that sends the math to a dummy backend worker to simulate that laggy, expensive feeling icekrieg wants." Investigation outcome: no real backend exists on this project (Supabase is the only server component, and a tile pyramid for the Mandelbrot set would be millions of tiles + GB of storage). The right architecture is purely client-side, but to give chat the requested theatrical experience, the FULL render pass is rerouted through a *simulated* tile farm when Server Mode is ON. **Worker pool**: 2-6 Web Workers (clamped by `navigator.hardwareConcurrency`), each rebranded as a "node" with a hostname from `compute-1.us-west` through `compute-6.ap-sg` and a base latency from 55 to 240 ms keyed to the fake region (US-West fastest, AP-Singapore slowest). The Mandelbrot kernel + smooth-coloring math is duplicated inside an inline `WORKER_SRC` string that's `Blob`'d into a worker URL on first activation; the active palette is shipped to each worker as a transferable Uint8ClampedArray on each LUT rebuild. **Tile farm dispatcher**: when Server Mode is on, the FULL pass calls `renderFullViaCluster()` which splits the viewport into `TILE_SIZE × TILE_SIZE` (192 CSS-px × DPR) tiles, queues them, and dispatches one tile per worker round-robin. Each tile gets: (1) an artificial **uplink delay** of `baseLatency*0.4 + jitter(40ms)` before the worker even starts crunching; (2) the actual escape-time render in the worker; (3) an artificial **downlink delay** of `baseLatency*0.4 + jitter(40ms)` before painting. Total per-tile latency = `~baseLatency + crunch_time`. The whole pass takes 2-8 seconds for a typical viewport, which is the "expensive feeling" chat asked for. **Cluster status panel** (top-right, only visible while Server Mode is on): amber-bordered glass card with a "fractal-farm cluster" title in Major Mono Display, a meta line showing live progress (`dispatching N tiles` → `12 / 24 tiles · 850ms wall` → `done · 2340ms · 24 tiles`), and one row per worker showing [status dot] [hostname] [current status] [last latency]. Status dot is purple (idle) / amber (busy) / green (done last tile). Row gets a `.busy` class while crunching, switches to `.done` after each tile and renders the recorded latency. **Tile-overlay markers**: a fixed-position overlay div spawns a `.tile-marker` rectangle for each tile being rendered — amber border with inset glow that oscillates via the `tileScan` keyframe (4→14px shadow over 0.6s). On tile completion the marker fades out over 350ms then removes itself. Lets users *see* the tiles arriving one at a time. **Stale-render protection**: every `dispatchToWorker` checks `epoch === currentRenderEpoch` at every step (before dispatch, before worker message bind, before delays fire, before paint). If the user pans/zooms mid-flight, in-flight tiles release their workers and abandon their markers gracefully. **Default state**: Server Mode is OFF on boot; the existing fast 3-pass progressive render is unchanged. Flipping the toggle to ON instantiates the worker pool (lazy — no cost when never enabled) and adds the cluster panel + tile markers to subsequent FULL passes. **No real backend**: every node, hostname, latency, and region tag is a label in JS. The actual compute happens in the user's browser. The cluster-foot row explicitly says `latency · payload · billing all simulated.` to make this clear to anyone reading.
- 2026-05-13: Shipped v1 — fullscreen interactive Mandelbrot set with smooth color transitions, progressive rendering, and 5 curated bookmark presets. Per chat: "create a fractal webapp with a never-ending, zoomable Mandelbrot set and smooth color transitions." **Renderer**: pure 2D canvas, single-threaded. The Mandelbrot escape-time kernel iterates `z_{n+1} = z_n² + c` with the standard `x² + y²` tracking + a single multiplication and 4 adds per iteration. Escape radius bumped to 256 (vs the usual 2) so smooth coloring near the boundary is genuinely smooth. **Smooth coloring** (the v1 design goal): when a point escapes, the integer iteration count `iter` is refined to a fractional value via `μ = iter + 1 - log₂(log|z|)`. This breaks up the visible iteration banding into a continuous gradient. A `Math.sqrt(μ / maxIter)` curve compresses the high-iteration tail so deep features have visible color variation. The result `t` is then wrapped `(t * 4) % 1` for the LUT lookup — the multiplier creates 4 color cycles across the available iteration range, which reads as flowing bands when you zoom. **Color palettes** (Iñigo Quílez cosine technique): `color(t) = a + b * cos(2π(c*t + d))` per RGB channel. 6 baked palettes — **cosmic** (purple/pink/cyan default), **inferno** (black/red/yellow/white), **ocean** (blue/cyan/teal), **forest** (green/yellow/orange), **mono** (grayscale), **aurora** (mint/purple/cyan). The active palette is baked into a 1024-entry Uint8ClampedArray LUT at app boot + every palette/shift change so per-pixel coloring is a single array indexing op (much faster than calling Math.cos 3× per pixel). A "shift" range slider cycles the palette by adding to `t` mod 1 — gives smooth animated re-coloring without re-rendering pixels (just rebuild the LUT). **Progressive rendering** — three-pass scheduler: on any view change, render LOW (1/4 canvas size, 96 max iter) immediately for instant feedback, then schedule MED (1/2 size, 384 iter) after 60ms, then FULL (device-pixel size, full user-set iter) after another 160ms. Each pass writes into its own offscreen canvas and is `drawImage`-blitted onto the display canvas with smoothing. Newer view changes cancel the queued passes via a render-epoch counter so a fast pan doesn't queue 10 useless full renders. Tiny pulsing dot in the bottom-right shows when a render is in flight. **Optimizations**: cardioid + period-2 bulb early-out checks before the iteration loop catch the largest interior regions in ~1 comparison each, saving roughly half the per-pixel work on the home view. **Pan + zoom**: mouse drag = pan, mouse wheel = zoom (toward cursor — the standard "Google Maps" pattern: factor 0.8 in / 1.25 out, point under cursor stays put after each step), double-click = zoom 2× toward cursor. Touch: 1-finger drag = pan, 2-finger pinch = zoom with the pinch midpoint as the anchor. Pinch math: capture starting distance + scale + complex coord under midpoint, then on each touchmove compute `scale = startScale * (startDist / currentDist)` and re-center so the same complex coord lands back under the (possibly moved) midpoint. Keyboard: `+`/`-` zoom, arrows pan by 15% of view scale, `R` reset, `S` save PNG. **Adaptive iterations**: when the zoom ratio exceeds 50×, max iterations auto-scales as `max(256, 256 + 60 * log₁₀(zoom))` and the iter slider updates to reflect it. User can still override via slider. **HUD** (top-right): re / im / zoom × / iter, all rebuilt after each render. Precision of re/im display grows with zoom — uses `Math.log10(zoom + 10)` to pick decimal places, so you actually see meaningful digits when you're 10⁸× deep. **Presets** (left side): home (-0.5, 0, scale 1.6), seahorse valley (-0.75, 0.1, scale 0.04), elephant valley (0.30, 0.01, scale 0.04), triple spiral (-0.7453, 0.1127, scale 6e-4), minibrot (-1.7497, 0, scale 8e-4). Each preset bumps the iteration count appropriately (384-512). **Save PNG**: forces a full-resolution render to the display canvas first, then `toDataURL` + click an anchor. **Aesthetic**: thin glassmorphic dark UI panels — `rgba(8,4,18,0.78)` with backdrop-blur and purple-tinted borders. Major Mono Display title ("mandelbrot·zoom" with purple `·`), Cormorant Garamond italic tagline ("drag · scroll · pinch · explore the edge of forever"), IBM Plex Mono for HUD/controls. Mobile (@720px): tagline hidden, control sizes shrunk, presets pushed up. Reduced-motion kills the progress-dot pulse. WCAG basics: semantic markup, role=status on HUD with aria-live, role=group on presets and controls, focus-visible accent outline, ≥44px touch targets on buttons (most are 35-44px — touchable but compact). Pollinations OG (flux seed 1618, deep Mandelbrot zoom in iridescent purple/cyan/gold).

## issues
- Float64 precision (≈10⁻¹⁵) limits the deepest zoom to roughly 10¹³×. Past that the pixelation becomes obvious and the fractal "freezes" because every pixel hits the same coord. Adding a perturbation-theory renderer would solve this but requires a reference orbit + complex perturbation math — significant work for a v2.
- Single-threaded — at full canvas resolution × 2× DPR × deep zoom × 2048 iter, the full-pass render can take a few hundred ms on slower devices. The progressive low/med passes keep interaction smooth but the full pass blocks the main thread. A Web Worker (or OffscreenCanvas with a worker) is the right v2 upgrade for buttery 60fps.
- Iteration auto-scaling on zoom is gentle (256 base + 60×log10(zoom)) so deep features may need user-overridden iter counts via the slider. The 2048 max is plenty for most beauty spots.
- The LUT uses `(t * 4) % 1` to multiply color cycles, which can look busy on very deep zooms where the smooth-iter values are bunched. A future option could expose the cycle count as a control.
- No anti-aliasing — at full res it's 1 sample per pixel. SSAA (e.g. 2×2 supersample on the FULL pass only) would noticeably improve fine detail but doubles render time.
- Touch pinch + simultaneous pan isn't perfect — the 2-finger pinch handler re-centers on the pinch midpoint but a small drift can occur during very fast multi-finger gestures. Acceptable for v1.
