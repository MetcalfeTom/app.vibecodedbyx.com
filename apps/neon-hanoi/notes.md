# neon-hanoi

## log
- 2026-05-26: Initial build. Classic Towers of Hanoi recursive solver with animated disk movement. **Algorithm**: pure-recursive `solveHanoi(n, from, to, via, out)` — the textbook 5-line definition that pushes each disk-move event into a flat sequence array up front. For N=3 produces 7 moves, N=5 produces 31, N=8 produces 255, all matching the closed-form 2^N − 1. Verified in JSDOM smoke test. **N range**: 3–10 disks (slider in HUD), default 5. N=10 produces 1023 moves which auto-plays in ~15s at 20× speed. **Per-move animation**: 3-phase pipeline — `lift` (0.16s, ease-in-out cubic raise to apex above pegs) → `travel` (0.26s, horizontal arc with small `sin(πt)*6` apex bulge for snappy motion feel) → `drop` (0.16s, ease-in-out fall onto destination peg's current top). Each disk is removed from its source peg's array at the start of `lift` (so resting disks shift down visually as if the piece "left the stack") and re-inserted at the end of `drop`. Total per-move time at 1× = 0.58s; speed multiplier scales `dt` directly in update (0.5×–20× range, default 2×). **Disk render**: per-disk hue ramps from 190° (cyan) for the smallest to 370° (warm-magenta) for the largest, given by `190 + ((disk-1)/(N-1)) * 180`. Width interpolates from 36 to 160 px (capped at 28% of canvas width). Rounded-rectangle body with vertical 3-stop HSL gradient (72%/52%/28% lightness), bright-hue stroke outline, 2px highlight strip at top, 3px shadow strip at bottom, disk number label centered (skipped when width <40 to avoid clutter). Lift state adds extra shadow blur for a "lifted" glow. **Peg render**: 8px-wide slate shaft + cyan stroke overlay with shadow blur, glowing top-cap dot, "A/B/C" label below platform. Platform = dark slab with bright cyan top edge. Background: void with magenta-pink horizon glow + cyan grid floor (14px horizontal lines + 12 perspective verticals fanning out toward the bottom). **HUD top**: 4 cards — N (violet), Move/Total (cyan), Optimal (gold, always = total), Speed (magenta). Below HUD: a live "step X: disk D → peg A ⟶ peg C" readout in lime that updates as each move begins. **Audio** (Web Audio synth): per-disk pitch via `1200 * (240/1200)^t` where t = (disk-1)/(N-1) — small disks high, big disks low. Lift = triangle pitch sweep up (skipped above 12× speed to prevent audio spam). Drop = triangle pitch sweep down + low sine thump for big disks + tiny noise burst. Complete = 5-note ascending C major arpeggio (523/659/784/1047/1318Hz triangle staggered 100ms). **Step mode**: Step button (or `S` / `→`) pauses auto-play and runs `startNextMove()` once, then a `stepUntilAnimationDone` flag in the update wrapper completes the current 3-phase animation before pausing — disk never gets stranded mid-air. **Play/pause**: Space toggle. **Reset**: rebuilds pegs + moves array from current N. Restarts automatically when Play is hit after completion. **Completion**: SOLVED banner pops with `cubic-bezier(0.34, 1.56, 0.64, 1)` overshoot + ascending arpeggio. **Defensive init pattern**: matchMedia + canvas + every handler in try/catch, global window.error renders magenta toast. **JSDOM smoke test**: zero errors, N=3→7 moves, N=5→31 moves, N=8→255 moves (formula verified), N input + speed slider + play toggle all wired. **WCAG**: rem units throughout, semantic main/h1, role=application + descriptive aria-label on canvas, aria-live move readout, aria-pressed play+mute, focus-visible 3px outline, ≥2.75rem interactive targets, prefers-reduced-motion damps dt ×0.8. **Aesthetic**: Audiowide title with cyan→magenta→gold gradient text-fill, Major Mono Display tagline, Share Tech Mono body, Rubik Mono One HUD numerals.

## issues
- The recursive solver is computed up front into a flat array — for N=10 that's 1023 entries (~30KB), trivial. If N ever goes above 15 (32K entries) we'd want lazy generation or iterative state.
- Lift sound is skipped above 12× speed because at 20× speed there'd be ~17 audio events per second, which sounds like a buzzsaw.
- "Sanity check" in startNextMove logs an error toast and pauses if the top-of-from disk doesn't match — should never fire if the recursive solver is correct, but it's a defensive guard against future bugs.
