# smooth-pong · notes

## log
- 2026-05-14: v1 — neon pong with acceleration-based paddle physics. Chat request: "create a simple pong game with neon colors and smooth paddle movement." Multiple older `pong-*` apps already exist in the repo (`neon-pong`, `neon-pong-blitz`, `fast-pong`, `pong-terminal`); this one differentiates by focusing on **buttery paddle motion**, not just instant velocity. **Court**: logical 800×500 rendered into a responsive canvas with DPR-aware scaling (`ctx.setTransform(...)`) so it stays sharp on retina. Aspect 8:5 box with violet border + violet glow + faint scanline overlay (1px every 3px). **Paddle physics**: per-paddle velocity + acceleration. Keyboard input (W/S or ↑/↓) applies `PADDLE_ACCEL = 4200 px/s²` accel; releasing keys multiplies velocity by `(1 - 7.5·dt)` for a soft decel — paddle coasts to a stop instead of stopping dead. Mouse/touch input switches to a critically-damped follow: a *desired* velocity is computed as `sign(diff) · min(|diff|·14, MAX)` and the paddle's current velocity eases toward that desired value via `(desired - currentV) · 12`. Either input style is butter-smooth and never twitchy. Max paddle speed 720 px/s. **Ball**: starts in center after a 0.8 s serve delay, launches at a random shallow angle (-0.3..0.3 rad) toward whoever lost the last point. Base speed 360 px/s. Each paddle hit adds 28 px/s (capped at 820) and reflects with an angle proportional to where on the paddle it struck (`hit ∈ [-1, 1]` → ±0.85 rad); paddle velocity is also injected into the ball's vy (×0.18) for English. Trail: last 16 positions rendered as expanding faded gold orbs. **AI**: smoothed tracking — when ball moves right (toward AI) it eases the paddle toward `ball.y + sin(t)·22 jitter`, when ball moves left (away) the AI idles by drifting near center via slow `sin(t·0.0007)·40`. AI max speed capped at 0.78× human max so it's beatable. **SFX (Web Audio)**: square-wave blips on wall bounce (280 Hz, .06 s, 0.14 gain), left-paddle hit (520 Hz, .07 s), right-paddle hit (440 Hz), descending sawtooth sweep on score (440→200 or 220→90 over .35 s), three-note triangle chord cascade on win (C5-E5-G5). Default unmuted; toggle in the HUD strip. Audio context lazily created and resumed on first user gesture (start button) so mobile autoplay rules are respected. **First to 7**: scores live in the header below the title in big Major Mono Display numerals, pink for left, cyan for right, with neon glow. Win overlay: shimmering "You win" / "Bot wins" big text + Play again button. **Aesthetic**: deep purple void with pink/cyan radial glows, Audiowide rainbow-shimmering title (animated background-clip gradient), Space Mono throughout. Hot-pink left paddle / cyan right paddle / gold ball; each paddle has a 22-blur shadow halo + inner white highlight stripe; ball has a yellow shadow glow + white-yellow gradient core. Dashed violet midline with glow. Particle bursts on wall bounce (5), paddle hit (8), score (18) — colored by which side. **Inputs**: W/S, ↑/↓, mouse-move over canvas, touch-drag on canvas, Space to start, P to pause. **WCAG basics**: canvas `role="application"` + summary aria-label, aria-live score, `aria-pressed` on mute toggle, `:focus-visible` outline, ≥2.75 rem button targets, `prefers-reduced-motion` disables title shimmer. Pollinations OG flux seed 70707.

## issues
- The DPR-aware scaling (`setTransform(W/canvas.width, ...)`) handles retina but doesn't re-update on `devicePixelRatio` changes mid-session (rare — happens if you drag a window between monitors). The `resize` listener covers viewport changes, not DPR-only changes.
- AI is intentionally beatable — its tracking is dampened and jittered. Currently calibrated for a casual win/lose ratio around 60% human win. Could expose a difficulty slider later.
