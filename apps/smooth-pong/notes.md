## log
- 2026-05-14: v2 — **green-on-black phosphor recolor + arrow-key fix + paddle swing rotation**. Chat reported jenn2's arrow keys weren't working on smooth-pong: root cause was the `mousemove` handler unconditionally overwriting `state.leftTargetY` and zeroing `keys.up/down` on *any* canvas-hovered mouse jiggle, so keyboard input could never beat passive mouse twitch. Fix: explicit `state.inputMode` ('keyboard' | 'pointer'), set on the most recent *intentional* input. Mousemove now only steals control when the mouse has moved more than 18 px in keyboard mode (cursor sitting still no longer hijacks anything); a real click/touch sets pointer mode explicitly. Update loop's mouse-follow branch now requires `inputMode === 'pointer'` to engage, so even if leftTargetY is set from earlier, keyboard input takes immediate priority. **Paddle swing (← / →)**: left/right arrows (or A/D) apply angular acceleration to the left paddle, rotating it up to ±0.6 rad (~34°). Released → spring restoration with damping pulls the paddle back to vertical. Max angular velocity 8 rad/s. Replaced the left paddle's AABB collision with proper OBB-vs-circle: transform ball into the paddle's local frame using `cos(-a)/sin(-a)`, clamp to the rectangle, check distance, reflect using the world-frame normal. Two physical English contributions on hit: linear paddle velocity (`leftV·0.18` into ball vy, classic spin) AND angular paddle velocity (`leftAngleV · SWING_ENGLISH / 30` into vy, plus `|leftAngleV|·4` added to vx for swing-driven pace). A safety branch handles the edge case where the OBB reflection would send the ball back into the left wall: it falls back to the original hit-position-based bounce angle but in the paddle's rotated frame. Hit SFX pitch scales with swing intensity (`460 + |angV|·18` Hz). Particle count also scales with swing strength. AI paddle stays AABB (no rotation). **Recolor**: full phosphor-terminal green-on-pure-black palette. CSS variables flipped from pink/cyan/violet/gold to `--green:#39ff8a`, `--green-deep:#15c060`, with black void. Body has only a soft central green radial glow + green scanline overlay using `mix-blend-mode:screen` (vs the old purple overlay). Title is solid green with stacked `text-shadow` glow layers (no rainbow shimmer). Score, eyebrow, paddles, ball, midline, particles all in the green band. Right paddle uses a slightly dimmer `#1ad470` for at-a-glance differentiation. Ball is white-green with a green halo. Paddle when rotated gets bright white top/bottom tip caps as orientation cues. Added a small `<kbd>`-styled help row below the HUD listing the new controls (W/S or ↑↓ move · ←→ swing · mouse/touch · P pause). Reset paddle angle on game start. Pollinations OG re-prompted for emerald CRT terminal vibe (flux seed still 70707).
- 2026-05-14: v1 — neon pong with acceleration-based paddle physics. Chat request: "create a simple pong game with neon colors and smooth paddle movement." Multiple older `pong-*` apps already exist in the repo (`neon-pong`, `neon-pong-blitz`, `fast-pong`, `pong-terminal`); this one differentiates by focusing on **buttery paddle motion**, not just instant velocity. **Court**: logical 800×500 rendered into a responsive canvas with DPR-aware scaling (`ctx.setTransform(...)`) so it stays sharp on retina. Aspect 8:5 box with violet border + violet glow + faint scanline overlay (1px every 3px). **Paddle physics**: per-paddle velocity + acceleration. Keyboard input (W/S or ↑/↓) applies `PADDLE_ACCEL = 4200 px/s²` accel; releasing keys multiplies velocity by `(1 - 7.5·dt)` for a soft decel — paddle coasts to a stop instead of stopping dead. Mouse/touch input switches to a critically-damped follow: a *desired* velocity is computed as `sign(diff) · min(|diff|·14, MAX)` and the paddle's current velocity eases toward that desired value via `(desired - currentV) · 12`. Either input style is butter-smooth and never twitchy. Max paddle speed 720 px/s. **Ball**: starts in center after a 0.8 s serve delay, launches at a random shallow angle (-0.3..0.3 rad) toward whoever lost the last point. Base speed 360 px/s. Each paddle hit adds 28 px/s (capped at 820) and reflects with an angle proportional to where on the paddle it struck (`hit ∈ [-1, 1]` → ±0.85 rad); paddle velocity is also injected into the ball's vy (×0.18) for English. Trail: last 16 positions rendered as expanding faded gold orbs. **AI**: smoothed tracking — when ball moves right (toward AI) it eases the paddle toward `ball.y + sin(t)·22 jitter`, when ball moves left (away) the AI idles by drifting near center via slow `sin(t·0.0007)·40`. AI max speed capped at 0.78× human max so it's beatable. **SFX (Web Audio)**: square-wave blips on wall bounce (280 Hz, .06 s, 0.14 gain), left-paddle hit (520 Hz, .07 s), right-paddle hit (440 Hz), descending sawtooth sweep on score (440→200 or 220→90 over .35 s), three-note triangle chord cascade on win (C5-E5-G5). Default unmuted; toggle in the HUD strip. Audio context lazily created and resumed on first user gesture (start button) so mobile autoplay rules are respected. **First to 7**: scores live in the header below the title in big Major Mono Display numerals, pink for left, cyan for right, with neon glow. Win overlay: shimmering "You win" / "Bot wins" big text + Play again button. **Aesthetic**: deep purple void with pink/cyan radial glows, Audiowide rainbow-shimmering title (animated background-clip gradient), Space Mono throughout. Hot-pink left paddle / cyan right paddle / gold ball; each paddle has a 22-blur shadow halo + inner white highlight stripe; ball has a yellow shadow glow + white-yellow gradient core. Dashed violet midline with glow. Particle bursts on wall bounce (5), paddle hit (8), score (18) — colored by which side. **Inputs**: W/S, ↑/↓, mouse-move over canvas, touch-drag on canvas, Space to start, P to pause. **WCAG basics**: canvas `role="application"` + summary aria-label, aria-live score, `aria-pressed` on mute toggle, `:focus-visible` outline, ≥2.75 rem button targets, `prefers-reduced-motion` disables title shimmer. ## issues
- The DPR-aware scaling (`setTransform(W/canvas.width, ...)`) handles retina but doesn't re-update on `devicePixelRatio` changes mid-session (rare — happens if you drag a window between monitors). The `resize` listener covers viewport changes, not DPR-only changes.
- AI is intentionally beatable — its tracking is dampened and jittered. Currently calibrated for a casual win/lose ratio around 60% human win. Could expose a difficulty slider later.
