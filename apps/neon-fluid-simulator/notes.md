# neon-fluid-simulator

## log
- 2026-05-01: Created. **Curl-noise neon ribbon fluid sim with cursor-driven field deformation.** Distinct from `neon-fluid` (which is dot+connection-line) — this app is **ribbon-trails** on a flowing curl-noise field. **Engine**: 400 default ribbons (configurable to 200/400/700/1000), each with a 28-position trail buffer (configurable 16/28/44/64). Per-frame: sample a cheap 2D pseudo-curl-noise field at the particle's position (2 layered sin/cos terms, perpendicular pairing for a curl-ish vector), inject `0.18 ×` of that into velocity. Cursor influence: within a 240px radius, particles get (1) a radial PULL toward cursor (`pullK = 0.50`, doubled while mouse is held), (2) a perpendicular SPIN around cursor (`spinK = 0.42`) — this is what gives the "fluid swirl" feel rather than a generic attractor, (3) a small drag from cursor's own velocity (`× 0.02 × falloff`). Falloff is linear from cursor center to radius edge. Velocity damped 4%/frame, capped at 6.0 px/frame, integrated, screen edges wrap. **Click → radial pulse**: a growing ring of force expanding to 380px over 0.7s; particles within 80px of the ring's leading edge get pushed outward proportional to `(1 - distToRing/80) × (1 - ageFrac) × 6`. Up to 8 simultaneous pulses queued. Pulse rings also drawn as fading expanding strokes for visual feedback. **Render**: two-canvas stack — `fluid` does the ribbon trails, `bg` does the cursor halo. Each frame the fluid canvas gets an 18%-alpha black overlay (`source-over`) for the fade-trail effect, then ribbons draw with `globalCompositeOperation = 'lighter'` for true additive neon glow. Each ribbon's trail rendered as a sequence of line segments (oldest → newest), with line width tapering 0.5 → 2.9 along the tail and alpha tapering 0 → palette glow factor. Hue per particle = `palette.hue[i % palLen] + per-particle offset (±25) + speed × 6` so faster ribbons skew warmer. Skip rendering across screen-wrap teleports. Hot core dot at the head adds an extra-bright pinprick. **Palettes** (cycle via P key or button): NEON (cyan/magenta/violet — default), EMBER (red/orange/pink), AURORA (green/teal/cyan/violet), BIOLUM (jellyfish-style cool greens + pinks), CANDY (hot pinks + amber + cyan). Each palette has a `glow` factor that scales the alpha curve. **Custom cursor**: native cursor hidden (`cursor:none`); bg canvas paints a radial-gradient halo (30px idle / 70px held) + small ring + tiny crosshair, all in palette tones, mix-blend implicit via additive bg. **Controls**: bottom-center pill row — palette / trail / ribbons / freeze / reset. SPACE freezes (button reads "❄ frozen" / "▶ flow"). P cycles palette. **Hint overlay**: "drag the cursor anywhere · click to pulse · hold SPACE to freeze" centered at start, fades out on first cursor move. **Aesthetic**: pure-black bg, Major Mono Display title with cyan+violet glow shadow, Syne Mono for buttons + hint, all letter-spaced uppercase. Mobile: `touch-action: none` on canvas, touch handlers route through the same setCursor + pulse logic. Pollinations OG.

## issues
- The curl-noise field is a cheap 2-layer sin/cos approximation, not a true Perlin/simplex curl. Looks fluid-ish but on close inspection has clear sin-band artifacts at low velocity. Acceptable for the visual style; a proper simplex implementation would cost a small library.
- 1000 ribbons × 64 trail = 64k segment-draws per frame — fine on desktop, may chug on low-end mobile. Per-particle render path is the bottleneck (lots of `beginPath/moveTo/lineTo/stroke` per segment for the per-segment width taper).
- Pulse rings can stack heavily if the user spam-clicks; capped at 8 in flight. Older ones get shifted out.
- No audio — would be nice to have a click thunk + ambient drone tied to cursor velocity. Future.
- Wrap-skip logic checks `Math.abs(dx) < W * 0.5 && Math.abs(dy) < H * 0.5` which is a heuristic; a 100% reliable approach would tag a "wrap" event in the trail buffer.
