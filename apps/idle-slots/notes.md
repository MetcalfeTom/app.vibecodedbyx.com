# idle-slots

## log
- 2026-05-01: **Win-condition audit — replaced hardcoded LINESETS table with a grid-walking generator that structurally cannot emit wrap-around lines.** Audit findings on the prior implementation: (a) **silent coverage gap** — `LINESETS = {6: [...], 9: [...]}` only had entries for N=6 and N=9, so at every other reel-count level (N=3, 4, 5, 7, 8) the line-detection loop ran over `[]` and awarded zero full-line, partial-line, or multi-line bonuses despite N=3 being the canonical 3-in-a-row slot machine layout AND the starting state of the game. (b) **No structural wrap-protection** — the hand-written tables happened to be correct, but a future edit adding e.g. `[2,3,4]` at N=6 (which crosses a row boundary from row 0 col 2 → row 1 col 0/1) would have looked like a horizontal line to the scorer and silently inflated payouts on impossible geometry. (c) **Asymmetric diagonals at non-square N** — N=7 and N=8 have a valid anti-diagonal `[2,4,6]` but the main diagonal `[0,4,8]` requires cell 8, which doesn't exist at N=8; naive generators would emit it. **Fix**: introduced `getLines(N)` that builds the same 2D grid `buildReels` paints (`cols = N≤4 ? N : 3`, `rows = ceil(N/cols)`), then walks rows/cols/diagonals only emitting lines whose every cell maps to a real reel index (`grid[r][c] !== null`). Wrap-around is impossible because the generator never crosses a row/col/diagonal boundary — it iterates the *physical* grid, not consecutive linear indices. Diagonals only fire on square grids (rows === cols && rows ≥ 3). Cached per-N. **Verified by fixture**: all 27 generated lines across N=3..9 pass a `isStraightLine` check (every cell on same row OR same col OR same +1/−1 slope) — no wraps. **Coverage gained**: N=3 → 1 line `[0,1,2]`; N=4 → 1 line of len 4 `[0,1,2,3]`; N=5 → 3 lines (1 full row + 2 cols); N=7 → 4 lines (2 rows + 1 col + 1 anti-diag); N=8 → 5 lines (2 rows + 2 cols + 1 anti-diag). Existing N=6 and N=9 sets match the old hardcoded ones byte-for-byte so established players see no regression — only the gap-fill and the structural guarantee.

## issues
- Idle progression rate is intentionally moderate at L0 (avg payout ~5c × ~0.4 spins/sec ≈ 2c/sec). PAYOUT L10 already pushes that to ~12c/sec; PAYOUT L20 lifts off into idle-game pacing.
- Wild symbols (🎰) currently count as their own symbol class for matching, not as multi-substitution wilds. Could be reworked to be true wilds for more strategy.
- The `prestigeGain()` formula uses `log2(runChips / 10000) - stardust`, so each successive prestige requires roughly 4× more runChips for the same stardust. If a player overshoots they only get a fraction of "fair" stardust — could be smoothed to log2 without subtraction.
- Floating chip pop is positioned in `reel-deck` coords using `bottom:50% + left:random(40-clientWidth-80)`; on very wide decks this concentrates pops near the center. Cosmetic.
- localStorage save fires on every upgrade buy + spin — fine on desktop, mild battery cost on mobile. Could debounce 250ms.
- No reset/wipe button beyond prestige; users who want a clean slate need devtools `localStorage.clear()`.
- Auto-spin doesn't fire while the tab is hidden (browser throttles setTimeout/setInterval). True idle progress while tabbed-out would need a "catch-up" calc on visibility resume.
