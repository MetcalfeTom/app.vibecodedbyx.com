## issues
- ~131 ingredients × 6 strip repeats = ~785 slot DOM nodes total. Tested fine on desktop; on low-end mobile the initial layout takes ~80ms but that's a one-time boot cost. Could lazy-build the strip on first spin for ultra-cheap initial render.
- The procedural blurbs cycle through 6 templates so after ~20 spins you start seeing repeats. More templates = more variety; trade-off vs file size.
- Reel lock state persists across spins but is NOT saved to localStorage — a refresh wipes the lock pattern. Could persist if users want to commit to "always almond milk" pinned.
- Glass fill hue blending is a circular average; some unusual combinations (e.g. red kefir base + green spinach booster) result in muddy brown gradients that don't read as vibrant. Acceptable since real such smoothies look exactly like that.
- Audio context starts on first user gesture (browser autoplay policy). The auto-spin at boot triggers SFX which the browser may suppress until the user clicks anywhere.
- Some emoji mappings are approximations (e.g. dates → 🌰, soursop → 🍃) since unicode emoji coverage of exotic fruits is uneven.
- The "alliterative override" pool is small per-fruit (3 entries each for 5 fruits); could expand if we want more name variety.
- Favorites' "OPEN" replays the saved recipe but doesn't restore the ingredient indices to literally the same items if the pools have been reshuffled — currently safe since pools are static, but if we ever shuffle pools at boot, the saved `idx` would point to wrong items.
