- 2026-05-18: v1.1 — **CORS workaround via codetabs proxy + cached detail modal + client-side arch filter**. Chat (awarpigeon): "found a CORS block on the v1 API, can we proxy it or find a workaround for app.sloppy.live?" then "use the codetabs proxy to fix the CORS block on the v1 API, and make sure to update the fetch logic to handle the new URL structure."
  - **Confirmed via curl** · neither v1 nor v2 Snapcraft API sends `Access-Control-Allow-*` headers, so any browser fetch from sloppy.live is rejected by CORS preflight. Tried snapcraft.io's own endpoints (login-redirect 302), corsproxy.io (paid), api.allorigins.win (522/down), thingproxy (DNS dead), proxy.cors.sh (429 anonymous), codetabs (HTTP 200 + `ACAO: *` + full v1 JSON). Codetabs wins.
  - **Proxy wiring** · new `proxied(url)` helper wraps any direct URL as `https://api.codetabs.com/v1/proxy/?quest=<encoded>`. `apiFetch()` always wraps before calling fetch. `_links.next.href` stays direct in state.nextUrl and gets re-wrapped on each pagination call.
  - **Custom headers dropped** · codetabs strips request headers it doesn't recognise. `/search` works without `X-Ubuntu-Series` / `X-Ubuntu-Architecture` (verified — returns 200 with full results). Removed both from `apiFetch`.
  - **Client-side arch filter** · since the server-side `X-Ubuntu-Architecture` filter no longer applies, new `filterByArch(results, arch)` filters each fetched page by `r.architecture.includes(arch)`. Status bar shows the pre-/post-filter delta: "showing 14 of 24 ... · amd64 (10 filtered out)".
  - **Detail modal uses cache** · `/details` and v2 `/info` BOTH require `X-Ubuntu-Series` and return 400 without it through the proxy. Realised the v1 search response already carries description + screenshots + license + base + version + confinement + binary_filesize + last_updated + architectures — every field the modal renders. Rewrote `openDetail(name)` to look up `state.results.find(s => s.name === name)` synchronously instead of fetching. Saves a round-trip and dodges the header problem entirely.
  - **Rate-limit handling** · `apiFetch` distinguishes 429 (rate limit), 403 (proxy denied), 502/503 (proxy down) from generic HTTP errors so the toast/error pane shows actionable messages.
  - **API info pill** updated to `api · v1 via proxy · <arch>` so chat can see at a glance how data is flowing.

## issues
- The snap store API doesn't return install counts or download numbers anywhere in the public response, so we can't show "1.2M installs"-style metrics that proprietary stores have.
- `ratings_average` is 0 for most snaps (the rating endpoint is separate + auth'd) — most cards won't show the ★ badge.
- The screenshot URLs sometimes have weird aspect ratios or are huge; we cap height at 12rem and let aspect breathe.
- v1 might eventually be deprecated. If it 410s or starts redirecting, swap `API_BASE` to `https://api.snapcraft.io/v2/snaps/find` and rewrite the per-result field access (v2 nests fields under `.snap` and uses media[] array for icons).
- No infinite scroll — explicit "load more" was the simpler choice. Easy to switch later if chat asks.
