## issues
- **Permanent + unmoderated** by design (chat asked for "permanent unhinged" + "no takedowns"). RLS only lets users delete their own rows; we have no admin delete UI. If griefing escalates, we'd need a soft-hide flag column + a moderator UI. Watch for this.
- **Wall is fixed-aspect to viewport** but tag positions are stored as 0-10000 (% × 100). Mobile-spray-then-desktop-view will reflow positions proportionally; tags placed near edges may clip slightly off-screen on narrower viewports. Acceptable.
- **z-index from `created_at`** uses `floor(ts/1000) % 1e6` which wraps every ~11 days; same-day tags layer correctly, but old vs new tags can flicker stacking after the wrap. Trade-off for not exposing serial id ordering.
- **Realtime spam** if many users post simultaneously: `sprayPshhh(0.7)` plays for every incoming insert. Could throttle the audio per-second to avoid a wall of psssh. Currently no throttle.
- **No offline queueing**: if user posts while disconnected the insert just fails with a red status. No retry buffer.
- **Drips** spawn relative to viewport at the moment of fresh-spawn; on resize they don't reposition (they're transient and self-remove anyway).
- **Author name is editable post-submit only by deleting + re-posting** — there's no row update UI. Acceptable for graffiti.
- **font/color stored as freetext strings** — DB will accept anything if a user crafts a malicious payload via direct supabase API. Renderer escapes text and treats color as inline style (potential CSS injection via author/font fields if user bypasses the UI). Mitigated by HTML-escape on text/author + class-allowlist for fonts (renderer ignores fonts not in `FONTS`). Color is rendered as `color: ${t.color}` — could allow `red; background:url(...)` injection from a crafty client. **Risk: low but non-zero.** Should sanitize hex on render.
